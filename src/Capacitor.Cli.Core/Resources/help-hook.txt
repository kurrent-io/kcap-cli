kcap hook — dispatch a vendor hook event

Usage: kcap hook --<vendor>

Description:
  Invoked by a coding agent's hook system, NOT by users directly. For most
  vendors it reads the hook payload as JSON from stdin (including
  `hook_event_name`) and forwards it to the kcap server; the vendor flag
  selects the per-agent dispatcher and the event is taken from the payload.

  Pi is the exception: it has no shell hooks, so its dispatch is argument-
  driven (no stdin payload) — see --pi below.

  The command is registered via 'kcap setup' or
  'kcap plugin install [--claude|--codex|--cursor|--copilot|--gemini|--pi]', which
  writes the appropriate hook configuration (or, for Pi, the extension) for the vendor.

Options:
  --claude    Dispatch the event as a Claude Code hook.
  --codex     Dispatch the event as a Codex CLI hook.
  --cursor    Dispatch the event as a Cursor IDE hook.
  --copilot   Dispatch the event as a GitHub Copilot CLI hook. Copilot
              payloads carry no event-name field, so the installer embeds
              it in the registered command (--event <name>).
  --gemini    Dispatch the event as a Gemini CLI hook. Self-routes on the
              payload's hook_event_name (SessionStart/SessionEnd/Notification).
  --pi        Dispatch a Pi (badlogic/pi-mono) lifecycle event. Pi has no
              shell hooks; the shipped kcap.ts extension invokes this with
              arguments instead of a stdin payload:
                kcap hook --pi --event <session-start|session-end> \
                  --file <session-file> [--cwd <cwd>] [--reason <reason>]
              The session id, cwd, and start time are read from the Pi
              session JSONL header.

Exit codes:
  0   Normal exit (including budget expiry, malformed payload, and
      disabled-session suppression).
  1   No vendor flag provided, missing hook_event_name, or server error.
