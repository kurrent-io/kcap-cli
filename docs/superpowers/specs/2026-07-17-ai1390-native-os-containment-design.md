# AI-1390 — Hosted-agent native OS containment: design spec

Parent: AI-1313 (spec `2026-07-16-ai1313-reviewer-capacity-reaping-design.md` §6.4(1), §10; Phase A kcap-server #1109 merged, Phase B kcap-cli #327). Repo: **kcap-cli only**. Sibling: AI-1391 (sequenced settlement — independent; neither blocks the other).

## 1. Problem

Phase B shipped the *managed* self-defense layer: durable PID records with exact start-identity, a kill-quarantine counted against admission (`EffectiveCount`), and a boot+heartbeat `OrphanReaper`. That layer is **recovery after the fact** — a child that outlives a dead daemon keeps running until a later daemon incarnation proves its identity and kills it. Three residual classes remain that only OS-level containment at spawn can close:

1. **Immediacy.** Between daemon death (crash/OOM/SIGKILL) and the next incarnation's reap, an orphaned reviewer keeps running — burning tokens, holding its worktree, and (until reaped) confusing "who holds the slot" diagnostics.
2. **The descendant-outlives-leader class** (surfaced by the #327 review, round 3). The shipped reaper signals the process group only *while the live leader anchors ownership* — once the leader is dead, a numeric pgid can no longer be proven ours (pid reuse), and "ambiguity never kills". A descendant that survived the leader's TERM exit before the group SIGKILL is therefore unreachable by the managed layer in the current epoch. (On Linux, *prior-epoch* descendants are still recoverable: they inherit the `KCAP_*` env markers, so the marker scan reaps them at the next boot/heartbeat. The unreachable residue is a **current-epoch descendant that left its process group** — rare, but real.)
3. **macOS has no automatic crash-survivor reaping at all** (Phase B finding — this *supersedes* the parent spec's §6.4(3) macOS story). macOS 26 redacts process env from same-user readers: both `ps -E` and `sysctl(KERN_PROCARGS2)` return no env, so (a) the env-marker scan is a designed no-op (`OrphanReaper.EnumeratePids()` returns `[]` on macOS), and (b) the record regime's Unix env guard can never confirm and always **spares**. A macOS daemon crash strands every hosted child until manual cleanup. Production is Linux, but every developer machine is macOS.

## 2. Shipped baseline this design builds on (kcap-cli #327)

- `UnixPtyProcess.Spawn` (`Pty/Unix/UnixPtyProcess.cs`): P/Invoke `forkpty`; the **child branch currently runs managed code** (pre-computed strings, then `execvp` P/Invoke). Fork-safety is by convention (pre-compute everything), not by construction.
- `ConPtyProcess` (`Pty/Windows/ConPtyProcess.cs:172-230`): already spawns via `STARTUPINFOEXW` + a **1-entry** `PROC_THREAD_ATTRIBUTE_LIST` (pseudoconsole).
- `Native/pty_shim.c` + csproj `CompilePtyShim` target: an existing native-shim precedent (`libpty_shim.dylib`), **compiled on macOS only** today; `LibraryImport("libpty_shim")`.
- `ProcessIdentity` (tri-state `MatchesTri`, zombie-aware `IsAlive`, ESRCH-only dead), `ProcessReaper` (proven-identity kills, group signal only while leader-alive-and-ours), `AgentPidRecordStore` (hashed filenames), `OrphanReaper` (epoch-guarded record pass + Linux env-marker scan), `AgentKillQuarantine` + `EffectiveCount`.
- Env markers `KCAP_AGENT_ID` / `KCAP_DAEMON_ID` / `KCAP_DAEMON_EPOCH` stamped into every child (inherited by descendants).

## 3. Goals / non-goals

**Goals.**
1. Immediacy is claimed **only where the OS + launch shape actually deliver it**, per the §4.5 matrix — never as a blanket statement: **Windows** — the agent + all descendants die with the daemon (job close), every death mode, unconditionally. **Linux** — the *leader* dies immediately while PDEATHSIG stays armed, **for a launch proven contained at initial exec on a ≥3.19 kernel**; uncontained classifications and post-exec disarming transitions fall back to the managed layer. **macOS** — no immediacy at all (no primitive); the deliverable is *eventual leader-level recovery* (M1-A). Everything the immediate path doesn't cover is a managed-layer (records/scan/quarantine) residual, enumerated in §4.5.
2. No false kills, ever — the parent's "ambiguity never kills" / anti-false-reap invariant survives **unweakened**: every kill this spec authorizes rests on an exact, kernel-assigned, non-wall-clock process-incarnation identity (or OS containment, where no identity decision exists at all). No probabilistic risk acceptance anywhere.
3. Restore macOS crash-survivor reaping to leader-level, eventual (next daemon boot/heartbeat) parity.
4. The managed layer (records/scan/quarantine) keeps working unchanged as the audit + backstop; wire protocol untouched.

**Non-goals.** Sequenced settlement / server consumers (AI-1391). cgroup-based Linux descendant containment (heavier operational footprint; revisit only if the group-escaped-descendant residue is ever observed in practice). Changing capacity accounting.

## 4. Design

### 4.1 W1 — Windows: creation-time Job Object (complete containment)

In `ConPtyProcess.Spawn`:
1. `CreateJobObjectW(NULL, NULL)` (anonymous, unnamed); `SetInformationJobObject(JobObjectExtendedLimitInformation)` with `JOB_OBJECT_LIMIT_KILL_ON_JOB_CLOSE`. Do **not** set breakaway flags — descendants cannot escape the job.
2. Grow the existing attribute list from 1 to **2 entries**: `PROC_THREAD_ATTRIBUTE_PSEUDOCONSOLE` (existing) + `PROC_THREAD_ATTRIBUTE_JOB_LIST` (the job handle). The child is a job member **at the instant it exists** — no suspended-then-assign window.
3. The job handle is owned by the `ConPtyProcess` instance (a `SafeHandle` field). Daemon death of any kind — clean dispose, crash, task-manager kill — closes the handle → the OS kills the child **and all descendants**.
4. Spawn-path failure after `CreateProcessW` returns (any subsequent step throws): `TerminateJobObject` + confirm death before rethrowing — the existing teardown/quarantine machinery then sees a confirmed-dead child.
5. Normal stop paths are unchanged (graceful stop → `Kill(entireProcessTree)`); the job is the backstop, not the primary kill.

On Windows, the PID-record layer becomes pure bookkeeping/diagnostics: there is no survivor class. Records are still written (uniform code path; they feed AI-1391's candidate reporting), and the boot record pass will simply find every record's process already dead.

**Floor + fail-closed rules (pinned).** `PROC_THREAD_ATTRIBUTE_JOB_LIST` requires Windows 10 / Server 2016+; ConPTY itself already requires Windows 10 1809+, so **1809 is the binding floor** — no new floor is introduced, but state it explicitly rather than citing Windows 8 nested-job support (which is about job nesting generally, not the attribute). Nested jobs cannot form when either job carries UI limits — our job sets **no UI limits** (only `KILL_ON_JOB_CLOSE`), and if the daemon runs inside an enclosing job that still prevents nesting, or `CreateJobObjectW`/`SetInformationJobObject`/the attribute update fails for any reason, the **launch fails** (fail closed with the API error surfaced via `LaunchFailed`) — a hosted agent is never silently spawned uncontained on Windows.

### 4.2 L1 — Linux: PDEATHSIG via a native spawn shim on a dedicated spawner thread

**(a) Native `pty_spawn` shim (new, in `pty_shim.c`).** Move the entire fork→exec child sequence out of managed code into one native function:

The parent does **all** resolution / classification / argv construction and hands `pty_spawn` a fully-formed **execution plan**; the shim only forks, arms PDEATHSIG, execs the plan, and runs the handshake — no path parsing, no argv building, no hidden state in the child.

```c
// The plan is a GENUINELY OPAQUE handle — the header exposes no fields; the shim
// owns everything it references (exec fd, paths, argv strings) for its whole life.
typedef struct pty_exec_plan pty_exec_plan;

// Startup capability probe (call once; result cached by the caller and passed into
// preflight — no hidden global). support: 1 = execveat(AT_EMPTY_PATH) usable, 0 = not.
// A test build may force 0 to exercise the <3.19 fallback without a legacy kernel.
int  pty_probe_execveat(void);   // uses the RAW syscall (see the kernel-floor rule); returns 1/0

// pty_preflight: resolve + classify, parent-side, pre-fork. It only opens/sniffs/
// preflights the target (O_RDONLY|O_CLOEXEC) when it is going to build an EXEC_FD
// plan (Linux AND execveat_supported); for macOS or a probe-disabled Linux kernel it
// copies the original absolute path + argv into an EXEC_PATH plan WITHOUT requiring a
// readable fd (an execute-only binary is valid for execve but fails O_RDONLY). On the
// EXEC_FD path, an inspection failure (open EACCES, unreadable header, shebang
// preflight error) that does NOT prove the path un-executable degrades to an
// EXEC_PATH-uncontained plan (loud warning) — never a launch failure; -1 is returned
// ONLY when the original path+argv plan itself cannot be constructed (e.g. the path
// does not exist). Never execs.
// envp is the SAME array later passed to pty_spawn — required here so a
// `#!/usr/bin/env NAME` shebang resolves NAME against the CHILD's PATH (env's
// actual semantics), not the daemon's ambient PATH (that would be the execvpe trap).
int  pty_preflight(const char* exe_abs_path, char* const orig_argv[],
                   char* const envp[], int execveat_supported, pty_exec_plan** out_plan);
int  pty_plan_contained(const pty_exec_plan* plan);   // 1 = proven non-privileged, 0 = uncontained (caller logs the warning)
void pty_plan_free(pty_exec_plan** plan);             // frees strings/argv, closes the plan's exec fd, sets *plan=NULL. Single-release; a second call with *plan==NULL is a no-op.

// pty_spawn: forkpty + child sequence + error-pipe handshake. 0 on success (out
// populated), -1 on failure (out.err_no/failed_step set). failed_step pinned enum:
// fork|prctl|parent_died|chdir|exec|handshake_timeout|cancelled. (Identity-capture
// failure is NON-fatal: the launch succeeds with start_identity=="".)
// start_identity is captured IN THE PARENT immediately after forkpty (before the
// child can be reaped) — see the capture-binding rule; empty => identity_unavailable.
typedef struct {
    pid_t pid; int master_fd; int err_no; int failed_step;
    char  start_identity[128];   // mac:{bootsessionuuid}:{p_uniqueid} / lx:{boot_id}:{starttime}; "" = uncapturable
} pty_spawn_result;
int pty_spawn(const pty_exec_plan* plan, char* const envp[], const char* cwd,
              unsigned short rows, unsigned short cols,
              pid_t expected_parent, int cancel_fd, pty_spawn_result* out);
```

**Capture-binding rule (pinned — closes the reuse-before-capture race).** The start-identity is captured **inside `pty_spawn`, in the parent, immediately after `forkpty` returns, before the pid is exposed to any reaper** (the forkpty child is not `waitpid`-ed by anything until `pty_spawn` returns it to managed code), and returned in `pty_spawn_result.start_identity`. It therefore identifies *the incarnation this spawn created*, not a possibly-recycled later occupant — a fast-exiting child cannot be reaped-and-replaced before capture because nothing has waited on it yet. Capture failure (private-ABI anomaly / `/proc` read error) yields `start_identity == ""` and `failed_step` is *not* set (the launch still succeeds) — the managed caller then writes an `identity_unavailable` record. The managed layer never re-captures identity for an already-spawned pid.

**Resource model (pinned — no per-launch leak across P/Invoke).** `envp`/`cancel_fd` are caller-owned and outlive the call. The opaque plan owns **exclusively** everything it references, including its exec fd; the managed caller wraps it in a `SafeHandle` whose release calls `pty_plan_free(&plan)` exactly once on every exit path that isn't a `pty_spawn` success — resolution error, queue cancellation, spawn failure — **and** after a `pty_spawn` success (the plan is spent once the child has exec'd; `master_fd` has transferred into `out`). An *uncontained* classification is **not** an early-free path: it still flows into `pty_spawn` and frees afterward like any launch. `cancel_fd`'s child copy is `O_CLOEXEC` (closed pre-exec) so it can't leak into the agent, like the error-pipe and exec fds. §5 adds fd- and native-heap leak assertions across a success/failure/cancel launch loop.

**The two modes have strictly separate argv/path contracts (pinned — this is the anti-double-shebang rule):**
- **`EXEC_FD`** (Linux, contained) — we bypass the kernel's shebang/PATH resolution entirely by exec-ing the interpreter fd, so the parent supplies the **rewritten** argv (rules below). Used *only* for a launch proven contained at initial exec.
- **`EXEC_PATH`** (macOS always; Linux uncontained/old-kernel fallback) — the kernel does its own normal resolution, so the plan carries the **ORIGINAL executable path + the ORIGINAL, UN-rewritten argv**. Never the resolved interpreter with a rewritten tail — that would make the kernel re-apply the shebang on top of our rewrite (`[interp, script, script, …]`). One rule, both platforms: EXEC_PATH = `{original path, original argv}`, and the OS resolves whatever it resolves (a macOS script's shebang, a Linux <3.19 script's shebang) exactly as `execvp` did before this change.

**argv rewrite (EXEC_FD only; parent-side; macOS never uses this path):**
- **Native ELF, no shebang:** `argv` = original argv unchanged; `mode = EXEC_FD`.
- **Direct shebang** `#!/abs/interp [optarg]`: `argv` = `[/abs/interp, optarg?, script_abs_path, orig_argv[1:]…]` — the single optional shebang arg IS kept (`#!/usr/bin/python -O` → `[python, -O, script, …]`); `argv[0]` = the resolved interpreter absolute path.
- **`/usr/bin/env NAME` form**: `NAME` is the command `env` resolves, **not** an arg to the interpreter — `argv` = `[<resolved NAME abs path>, script_abs_path, orig_argv[1:]…]` (NEVER `node node script`). `NAME` is resolved against the **child's `PATH`** (from the `envp` passed to `pty_preflight`, the same array `pty_spawn` receives), closing the "resolve against the daemon's ambient PATH" (`execvpe`) trap. To keep resolution correct *by construction* — without threading the child cwd into the parent and reproducing cwd-relative lookup — the contained fast-path requires an **absolute-only child `PATH`**: if `PATH` is unset, use the string returned by `confstr(_CS_PATH)` **at runtime, verbatim (preserving its directory order)** — it is configuration/platform-dependent (typically all-absolute), *not* a fixed literal; if that string itself contains a non-absolute element, treat it like any relative-PATH case → uncontained; but if the child `PATH` contains **any empty or non-absolute element** (`""`, `.`, `bin`, …) — which `env` would resolve against the child's post-`chdir` cwd, a directory the parent-side preflight deliberately doesn't have — the plan is classified **uncontained** (`EXEC_PATH` original). That is safe and correct: at runtime the kernel/`env` resolves those relative components against the real child cwd natively; we simply forgo containment for that pathological launcher shape rather than risk preflighting the wrong inode. With an absolute-only `PATH`, the preflighted-and-exec'd interpreter is exactly the one the child's env would have selected. If `NAME` isn't found in an absolute-only child `PATH`, also classify **uncontained** (`env` would fail identically at runtime, consistently). The `envp`/`PATH` snapshot read at preflight is the same array passed to `pty_spawn` and is **immutable between the two calls** (the caller must not mutate it).
- **`env` with extra tokens** (`-S`, `VAR=val`, multiple args), **a chain deeper than script→interpreter**, or **an unresolvable shebang** → not rewritten → `contained = 0`, `mode = EXEC_PATH` with the **original** path + argv (the fallback contract above; the kernel/`env` handles it, uncontained). `cancel_fd` sits in the parent's handshake `poll` set alongside the error pipe, so **shutdown cancellation is deliverable to the blocking P/Invoke** (managed shutdown writes a byte; the shim kills + reaps the pending child and returns `failed_step = cancelled`) — no hidden globals, no native logging; the managed caller renders `failed_step`/`contained` into structured logs + the uncontained warning.

**Executable resolution is pre-fork, in the parent (pinned).** The managed caller resolves the launcher's `CliPath` against `PATH` *before* calling `pty_preflight`, which opens + classifies the object the child will exec; the child then execs **that fd** — `execveat(fd, "", …, AT_EMPTY_PATH)` on Linux — or, on macOS, `execve` of the absolute path (no fd-exec primitive there, and no PDEATHSIG to guard). Both are async-signal-safe. `execvpe` is rejected: it is a GNU extension absent from the macOS SDK, is not async-signal-safe, and (the semantic trap) glibc's `execvpe` resolves against the *caller's* `PATH`, not the supplied `envp`. **Behavior change, accepted and documented:** the current `execvp` path's ENOEXEC shell-fallback (interpreting a script without a shebang via `sh`) is *not* preserved — a shebang-less script now fails the launch with `ENOEXEC`, surfaced like any launch failure.

Child-side sequence (async-signal-safe calls only, no allocation, no managed re-entry — it just executes `plan`):
**(parent, immediately after `forkpty`, before returning/reaping: capture `start_identity` — the capture-binding rule.)** Then the child runs: `prctl(PR_SET_PDEATHSIG, SIGKILL)` *(Linux only)* → `if (getppid() != expected_parent) { write parent_died to the error pipe; raise(SIGKILL); }` → `chdir(cwd)` → exec per `plan.mode`: `EXEC_FD` → Linux `syscall(SYS_execveat, plan.exec_fd, "", plan.argv, envp, AT_EMPTY_PATH)` (the **raw** syscall — no glibc-2.34 wrapper floor, no `/proc` dependency, async-signal-safe; an `O_CLOEXEC` fd is fine because the close happens *on successful exec*); `EXEC_PATH` → `execve(plan.exec_path, plan.argv, envp)` (macOS always; Linux uncontained fallback) → on failure write the failing step + `errno` to the error pipe → `_exit(127)`.

**Why the plan execs the interpreter fd, not the script fd (closes the shebang TOCTOU).** `execveat` of a *script* fd would make the kernel re-resolve the interpreter **by pathname** (and `/usr/bin/env` would re-resolve its target again) — the preflighted object would *not* be what runs. So `pty_preflight` resolves the shebang **once, pre-fork**, opens + preflights the **final interpreter ELF**, and puts *that* fd + the rewritten argv (per the argv rules above) in the plan; the child execs the fd directly, bypassing kernel shebang resolution and `env`. The exec'd object is therefore exactly the preflighted inode, for binaries and scripts alike (true for `claude`, an env-shebanged script; `codex` is a native ELF). The *script's contents* remain runtime data read by the interpreter — mutable like any source file, outside containment's scope (the kernel ignores setuid/file-caps on scripts anyway; PDEATHSIG-clearing is a property of the exec'd binary = the interpreter we checked). A chain deeper than one script→interpreter level, or an unresolvable/multi-token shebang, is classified **uncontained** (`EXEC_PATH` of the original path, loud-warning).

**Child-failure reporting (error pipe, pinned).** `pty_spawn` creates a self-pipe before forking, with **both fds flagged close-on-exec before the fork**. Descriptor ownership: the parent closes the write end immediately after `forkpty` returns; the child closes the read end as its first act. **Every deliberate child-side abort writes a distinguished `{step, errno}` record first** — including the `getppid` parent-death self-kill (finding: a silent `raise(SIGKILL)` would close the pipe and read as EOF/success to a live parent that passed a wrong `expected_parent`). Parent read semantics:
- **bytes** → child failed at a shim-controlled step → `waitpid` the child and return `-1` with the child's step+errno.
- **EOF** → the exec replaced the image (CLOEXEC closed the pipe) → return the pid. *Claim scope, narrowed:* EOF proves no **shim-controlled step** failed. An asynchronous **external** SIGKILL inside the window is indistinguishable from exec-then-immediate-death — in both cases the child is *dead*, the parent's normal exit-noticing path (read-loop EOF / `HasExited`) classifies it exactly like any agent that died at startup, and the record layer sees a confirmed-dead pid. So the invariant is: `pty_spawn` never reports success for a child that failed a step the shim controls, and never leaves a live-but-unexeced child behind.
- **bounded wait** → the parent read is a `poll`+read with a deadline (30 s) — a stopped/hung pre-exec child or a hung filesystem cannot wedge the (sole, daemon-lifetime) spawner thread: on deadline or shutdown-cancellation the parent kills + reaps the child and returns `-1`/`ETIMEDOUT`.

macOS portability of the CLOEXEC pipe (finding): the macOS SDK has no `pipe2(O_CLOEXEC)`, so `pipe()` + `fcntl(FD_CLOEXEC)` is non-atomic against concurrent forks *elsewhere in the process* (e.g. `Process.Start` for auxiliary tools). Consequence analysis, pinned: a racing fork can only *inherit the write end*, which **delays EOF** until that unrelated child exits — it cannot fabricate bytes or corrupt the protocol — and the bounded read deadline converts an over-long delay into the safe kill+reap+`ETIMEDOUT` path. The two syscalls run back-to-back inside the shim, so the window is negligible; the deadline makes even the pathological case safe rather than wedging.

A parent-side failure after fork but before returning (pipe read error, allocation) kills + reaps the child before returning `-1` — `pty_spawn` never returns failure while leaving an unobserved live child. This closes the current `execvp`-path gap where a child that failed to exec is indistinguishable from a running agent until its exit is noticed asynchronously.

The `getppid` re-check closes the fork→prctl death window: a daemon that died in between reparents the child (to init/subreaper), the check fails, the child self-kills. `expected_parent` is captured pre-fork by the caller. This also *removes* the existing managed-child-branch fork-safety-by-convention: after L1, no managed code ever runs between fork and exec on any Unix.

**PDEATHSIG survives exec only for non-privileged executables — preflight is fd-bound to the executed object (pinned).** The kernel *clears* the parent-death signal when `execve` runs a set-user-ID / set-group-ID binary or one carrying file capabilities, and the pre-exec re-check cannot repair a flag cleared *by* the exec. A path-based `stat`-then-`execve` check would be TOCTOU-broken (the file can be replaced between check and exec), so on Linux the check and the exec are bound to **the same open file**: `pty_preflight` opens the object `O_RDONLY|O_CLOEXEC`, preflights via `fstat` (setuid/setgid bits) + `fgetxattr(fd, "security.capability")` on that fd, and the child executes **that fd** via `execveat(fd, "", …, AT_EMPTY_PATH)` — what was checked is what runs, by construction, with the script→interpreter case handled by the plan (the fd in the plan is always the fd that execs). Classification is fail-closed: only a clean "no bits + no capability xattr" (`ENODATA`, or `ENOTSUP` from a filesystem that cannot carry the xattr) proves non-privileged; **any other `fgetxattr`/`fstat` outcome classifies the launch uncontained** (never a false proof).

**Kernel floor (pinned — real runtime probe via the raw syscall).** `execveat(AT_EMPTY_PATH)` needs a **Linux ≥ 3.19 kernel**; the glibc `execveat` *wrapper* only appeared in glibc 2.34, so both the probe and the child use the **raw syscall** `syscall(SYS_execveat, …)` — this keeps the floor a kernel floor, with no accidental userspace/glibc floor. `pty_probe_execveat()` runs once at startup: `syscall(SYS_execveat, -1, "", NULL, NULL, AT_EMPTY_PATH)` and reads `errno` — **supported** iff `errno == EBADF` (the syscall exists and reached fd validation of our deliberately-bogus `-1`); **unsupported** on `ENOSYS`; **and — fail-safe — every other errno** (`EINVAL` invalid-flags, `EPERM`/policy filters like seccomp, or anything unrecognized) is treated as **unsupported** so classification can never be left undefined. The cached boolean is passed into `pty_preflight(…, execveat_supported, …)`; unsupported ⇒ it never emits `EXEC_FD` — a would-be-contained launch is built `EXEC_PATH`-uncontained (loud warning) at *plan-construction* time, so the child never discovers `ENOSYS` post-fork. The probe result is also the **test seam**: a test forces it to 0 to exercise the `<3.19`/no-fd-exec fallback without a legacy kernel. A documented-minimum note is explicitly not relied on.

**Post-exec transitions:** an agent that later re-execs a privileged binary or changes credentials sheds PDEATHSIG by kernel rule — outside any spawn-time check's power. The Linux containment claim is therefore pinned narrowly as **"immediate leader death while PDEATHSIG remains armed, for a launch proven contained at initial exec on a ≥3.19 kernel"** — *not* unconditional. Every other case (privileged/unreadable/deep-shebang/multi-token-env → uncontained; older kernel → uncontained; post-exec privileged re-exec or credential change → PDEATHSIG shed) is a **managed-layer-only residual** (records/scan), exactly as the §4.5 matrix enumerates. An uncontained classification does **not** fail the launch (no real hosted-agent CLI is privileged — refusing would be gratuitous): it logs loudly and proceeds with the record/scan layers as its only coverage, the same posture macOS has.

**(b) Dedicated daemon-lifetime spawner thread.** `PR_SET_PDEATHSIG` fires when the creating **thread** dies, not the process — a pool thread retiring would SIGKILL every healthy reviewer it spawned. All Linux `pty_spawn` calls therefore run on **one dedicated native thread** created at daemon start and parked between spawns (a simple `BlockingCollection<SpawnRequest>` loop in a `Thread { IsBackground = false }`), never a pool thread. Failure policy, pinned: **unexpected spawner-thread termination fail-fasts the daemon process** (`Environment.FailFast`) — children then die *with* the daemon, which is exactly the semantic PDEATHSIG advertises — rather than a half-broken daemon keeping agents alive that a thread death has silently disarmed. Normal shutdown stops agents first, then retires the thread.

**(c) Scope + build.** PDEATHSIG kills only the immediate child (the forkpty session leader). Descendants: PTY EOF on leader death + the inherited-env marker scan (prior-epoch) remain the coverage; the group-escaped current-epoch descendant stays the accepted, logged residual (§1.2). The records/scan layers **stay fully active on Linux** — L1 covers *only* the leader, *only* for a contained-at-initial-exec launch (per the pinned claim above), so layers 2–3 are the backstop for everything else, not retired. To avoid two divergent Unix spawn paths, **macOS also routes through `pty_spawn`** (same shim, `prctl`/`execveat` compiled out under `#ifdef __linux__`, `execve` on macOS) — one native child sequence on both Unixes, fork-safety by construction everywhere.

**Build/packaging (RID-safe, pinned).** The shim must be compiled **per RID on that RID's native runner** — exactly like the existing 5-platform AOT binary matrix (a `cc` on the build host emits the host's architecture; cross-publishing `linux-arm64` from x64 would silently package an x64 `.so`). Output paths are RID-isolated (`$(OutputPath)$(RuntimeIdentifier)/` or equivalent), never a shared `$(OutputPath)` that collides across RID builds. Acceptance is stronger than "a `.so` exists": per published artifact, (i) verify the binary's machine type matches the RID (ELF `e_machine` / Mach-O cputype), and (ii) a smoke test loads the shim and calls `pty_spawn` on a trivial command — run as part of the AOT publish checks in CI.

### 4.3 M1 — macOS posture (decision)

No OS primitive exists (no PDEATHSIG, no job objects; `kqueue EVFILT_PROC` requires a live watcher — i.e. the dead daemon). **M1-A is not containment** — it is *eventual, leader-level recovery*: a survivor lives until a later daemon incarnation's boot/heartbeat record pass, and indefinitely if no incarnation starts. That scoping is deliberate (Goal 1). Two options:

- **Option A (recommended, do now): a real macOS incarnation identity — `p_uniqueid`.** The wall-clock start stamp is the wrong identity primitive on macOS, so don't build on it: the kernel already assigns every process a **boot-scoped, monotonically-allocated 64-bit unique process id** — `proc_pidinfo(pid, PROC_PIDUNIQIDENTIFIERINFO).p_uniqueid` — readable for same-user processes (unlike env). Mint a new token scheme, structurally identical to Linux's `lx:{boot_id}:{starttime}`:

  `mac:{kern.bootsessionuuid}:{p_uniqueid}`

  — the boot-session UUID (a freely readable `sysctl`) scopes it exactly as `boot_id` does on Linux, and `p_uniqueid` is a kernel counter, monotonically assigned per process creation, **never reused within a boot timeline** and **not wall-clock**. `ProcessStartToken` gains the `mac:` scheme (capture + compare); the existing tri-state rules apply unchanged: same scheme + same value = ours, same scheme + different value = conclusively recycled, different scheme / unreadable / anomalous = **spare**.

  **Private-ABI dependency, pinned as a conscious decision.** `PROC_PIDUNIQIDENTIFIERINFO` (flavor 17) and `struct proc_uniqidentifierinfo` are `#ifdef PRIVATE` in the public SDK's `sys/proc_info.h` — `proc_pidinfo` itself is public, the flavor is not. M1-A **vendors** the constant and the struct layout (fixed, from the xnu open-source tree; a `static_assert` pins the expected size) rather than pretending an SDK header exists. Fail-safe rules make ABI drift *spare-shaped, never kill-shaped*: the capture accepts the result only if the call returns exactly the expected byte count **and** `p_uniqueid != 0`; any `EINVAL`, short/over-size read, zero id, or future removal → the token is **uncapturable**.

**Uncapturable-identity record contract (pinned — supersedes "record without a `mac:` token").** The inherited schema's `start_identity` is a real field and `PersistPidRecordOrThrow` is fail-closed, so "just write it null" is under-specified. M1-A adds a small distinguisher: a record carries `start_identity` **plus** an `identity_kind` enum ∈ {`present`, `identity_unavailable`}. On uncapturable identity (`pty_spawn_result.start_identity == ""`) the daemon writes `identity_kind = identity_unavailable` with an empty token — a *deliberate, well-formed* record, distinct from a malformed/corrupt one (which `AgentPidRecordStore` still quarantines as `.corrupt`).

**Backward-compatible deserialization (pinned — the rollout contract depends on it).** Every pre-M1-A record (Windows/Linux/macOS) has a nonempty `start_identity` and **no** `identity_kind`. The load rule: **missing `identity_kind` + nonempty `start_identity` ⇒ decode as `present`** (so old Linux/Windows records keep working and old macOS `tk:` records load and become `legacy_unresolvable` via the cross-scheme rule — they are *not* dropped or quarantined as corrupt). The new field is therefore **optional-with-a-default on read**, never required. Inconsistent *new* combinations are the only rejected shapes — `present` + empty token, or `identity_unavailable` + nonempty token → quarantined `.corrupt` (a real corruption signal, not a legacy record). The §5 upgrade test **deserializes the exact old JSON with `identity_kind` absent** (not a current model object hand-labeled "legacy") and asserts it loads as `present`. Behavior: the **launch proceeds** (the child is spawned and healthy — failing an otherwise-fine launch because a private-ABI probe hiccuped would be worse than the residual); the record is retained and, on every reap pass, an `identity_unavailable` record can *never* authorize a **record-pass** kill (no token to match). Its automated recoverability then splits by platform, because the env-marker scan is a **second, independent** kill authority that reads the *live process's own* `KCAP_*` triple — not the record's `start_identity`:
- **Linux:** an `identity_unavailable` record does **NOT** suppress marker-scan candidacy (pinned) — a prior-incarnation such process still carries its `KCAP_AGENT_ID`/`KCAP_DAEMON_ID`/`KCAP_DAEMON_EPOCH`, so the env-marker scan can reap it like any recordless survivor. But the record stays **pending (unresolved) until a marker read positively confirms it**, with a strict lifecycle to avoid re-introducing a PID race: (i) while pending it **blocks `StartupReapComplete`** exactly like any unresolved candidate — a failed/unreadable/not-yet-run marker scan leaves it pending, never "absent" (so AI-1391 can't treat omission as absence and relaunch alongside the survivor); (ii) the marker scan authorizes the kill by the *live env triple*, reads `KCAP_AGENT_ID` from that same live process, and on **confirmed death** deletes the matching `identity_unavailable` record **in the same operation** — positive, PID-independent resolution, so a reused PID between the kill and any later record pass can never make an identity-less record act on an unrelated process (the record is already gone) nor linger forever. Deletion is therefore *never* deferred to a "later record pass keyed on the numeric pid".
- **macOS:** there is no marker scan (env redacted), so an `identity_unavailable` record is genuinely **`identity_unresolvable` / manual-only** — logged each sweep with pid + age, resolved by the same manual-verify-and-kill operator contract as `legacy_unresolvable` (after which the dead pid deletes the record normally).

For AI-1391, an `identity_unavailable` record blocks `StartupReapComplete` on **both** platforms while pending; the difference is only how it resolves (Linux: automatically, via a confirmed marker kill; macOS: manual). A launch is **never** failed closed for this. §5 tests: a **forced Linux capture-failure → daemon-restart** case with two arms — (a) marker read unreadable/failed ⇒ record stays pending, completion stays false, no false absence; (b) marker read succeeds ⇒ confirmed kill deletes the record atomically, and a **forced PID reuse between the kill and a later sweep** proves the record is already gone (never acts on the new occupant); and a macOS case persisting + reloading an actual `identity_unavailable` record (not a mocked "spare") asserting `identity_unresolvable` + the manual-kill deletion path. A **runtime capture smoke test runs in the per-RID macOS packaging checks** (capture the current process's own id and a child's, assert non-zero and distinct) — availability is proven by execution, not by unit-mocked round-trips. Minimum floor: macOS 12+ (the flavor predates it by years); both `arm64` and `x86_64`.

  **Uniqueness scope, honestly stated (supersedes "by construction" in any earlier draft):** `p_uniqueid` comes from a boot-local kernel counter, and `kern.bootsessionuuid` identifies a *boot*, not a *timeline* — a **full-VM snapshot restore rewinds both**. The uniqueness proof therefore holds for **one non-rewound boot timeline whose daemon state dir is snapshotted/restored atomically with the kernel** — which is exactly the physical reality of the actual macOS fleet (developer laptops: the state dir lives on the same APFS volume as the OS; a restore rewinds records and counter together, keeping them consistent). The documented exception is a macOS VM whose kcap state dir lives on storage *not* rolled back with the VM: there a discarded-timeline record could exactly match a post-restore process. Pinned handling: this is an **unsupported deployment** (stated in the daemon docs next to the state-dir description); no probabilistic hand-wave — the invariant is scoped, the scope is enforced by documentation + the fleet's actual shape, and operators running snapshot-restored macOS VMs with external state dirs must move the state dir inside the snapshot boundary.

  **Precision honesty (supersedes the parent's phrasing and Phase B's comment wording):** the shipped macOS `tk:{StartTime.Ticks}` token derives from the kernel's `pbi_start_tvsec/tvusec` — a **microsecond-resolution wall-clock** stamp; the `Ticks` representation adds no resolution, and wall-clock identity is collision-prone under clock steps. That is exactly why M1-A replaces it rather than adding compensating factors. `tk:` remains in use on Windows, where the record pass is moot anyway (job containment).

  **Legacy `tk:` records do NOT age out on their own — explicit contract (pinned).** A record written by a pre-M1-A daemon compares cross-scheme → `null` → spared, *every* pass, forever: a live headless survivor behind such a record is **permanently unreapable by automation** (the stop fallback reads the same record; server healing has no positive identity either). Rollout residual, handled explicitly rather than implied away: the record pass classifies these as `legacy_unresolvable` and logs each sweep with pid + record age; the operator contract is *manually verify and kill the pid* — after which the next sweep observes the pid dead (`IsAlive` false → confirmed-gone) and deletes the record normally. For AI-1391, a `legacy_unresolvable` candidate keeps `StartupReapComplete` false and is carried in the blocked-candidate detail, so the condition is visible server-side, not silent. The §5 tests include a live-legacy-survivor upgrade case (loud log + record retained + post-manual-kill deletion), not just token-comparison units.
- **Option B (deferred, only if the residue matters in practice): guardian process.** The parent spec §10 sketch — a tiny intermediary that only execs the agent after the record is durable and self-kills its process group on daemon-pipe EOF. Closes immediacy + descendants on macOS, at the cost of an extra process per agent and a new native artifact. Not worth it while macOS is dev-only; the spec records it so the decision is conscious.

M1-A also does **not** restore the macOS env-marker scan — a **recordless** survivor stays permanently unreachable there. The record write is fail-closed since #327 (`PersistPidRecordOrThrow`), so the recordless window is `forkpty`-return → record rename; that window is *typically* sub-millisecond but **unbounded** (the parent can be descheduled or the disk can stall arbitrarily long, and SIGKILL in the window leaves no record — "fail-closed write" cannot cover a death that prevents the write). Accepted, log-only residual; the §5 macOS test list includes a crash-injection case in this window so the residual is *documented by a test*, never mistaken for coverage.

### 4.4 Interaction with the shipped managed layer

- `EffectiveCount`, quarantine, teardown, and stop paths are untouched — containment only changes *what the OS does on daemon death* and *what the boot pass finds* (nothing, on Windows; only genuinely stuck processes elsewhere).
- The `OrphanReaper` record pass stays as-is (epoch-guarded); on Windows it becomes a no-op-in-practice audit that deletes already-dead records.
- No wire/protocol change; nothing for the server. (AI-1391's `StartupReapComplete` semantics get *simpler* on contained platforms — boot discovery trivially completes — but that is AI-1391's concern.)

## 5. Testing (kcap-cli unit suite; isolated dummy processes only — no live daemon/flows)

- **Windows (CI windows-latest):** creation-time association — crash-injection immediately after `CreateProcessW` leaves no survivor; kill-on-close with a dummy child + grandchild (both die on job-handle close); spawn-failure after create → `TerminateJobObject` + confirmed-dead; **breakaway actually attempted and denied** — a child that calls `CreateProcess(CREATE_BREAKAWAY_FROM_JOB)` fails (mere grandchild membership doesn't prove escape is impossible); a mini-daemon host **already inside an outer job** still nests successfully (and the fail-closed path fires when nesting is genuinely prevented); job-creation/assignment failure → launch fails with the API error (never a silent uncontained spawn).
- **Linux (CI ubuntu-latest):** PDEATHSIG kills the child when the *spawner-thread's process* dies (fork a disposable "mini-daemon" test host, kill it, assert the dummy dies); the `getppid` re-check writes `parent_died` to the error pipe **and** self-kills — `pty_spawn` returns `-1` with that step, never a false success (simulated via wrong `expected_parent`); an agent **survives** unrelated thread churn while the daemon lives (spawner-thread invariant); unexpected spawner-thread exit fail-fasts (process-isolated test host); the post-fork path is native-only (asserted structurally: the managed child-branch code is deleted); **preflight vs child failure, kept distinct** — a **missing/nonexistent original path** is a *preflight* failure (`pty_preflight` → `-1`, **no child forked**), asserted separately; a **child-side** failure (`failed_step=exec` with the child's errno + a reaped child, no zombie/phantom) is exercised by building a valid `EXEC_PATH` plan and then removing/swapping its path (or a post-fork missing dependency) between preflight and spawn — plus bad-cwd (`failed_step=chdir`) the same way; a **pre-exec-stopped child** (`SIGSTOP` injected) hits the read deadline → killed + reaped + `ETIMEDOUT`; **shutdown during the handshake** cancels the same way; **fd-bound preflight + interpreter-fd exec** — a setuid dummy classifies uncontained (warned, launch proceeds); an unreadable-xattr fixture classifies uncontained (never a false non-privileged proof); an env-shebang script execs its **preflighted interpreter inode** — asserted via `/proc/{child}/exe` inode equality **under a path-swap race** (the interpreter path is replaced between preflight and spawn; the child still runs the checked inode); **env resolves against the CHILD PATH, not the daemon's** — with the daemon's ambient `PATH` and the supplied child `envp` `PATH` (absolute-only) pointing at *different* `NAME` binaries, the contained plan preflights + execs the one the **child envp** selects; **empty/relative child PATH component → uncontained** (asserted with daemon-cwd and child-cwd holding different `NAME` binaries and a `.`/empty child `PATH` element: the plan is `EXEC_PATH`-uncontained and runs correctly via the kernel/`env`'s own cwd-relative resolution, never a parent-preflighted wrong inode); unset child `PATH` → selection matches the runtime `confstr(_CS_PATH)` string verbatim (the test compares against that queried value + its order, not a hardcoded `/usr/bin:/bin`); the child has no leaked executable/pipe fds (O_CLOEXEC policy asserted); a two-level script chain classifies uncontained; ENOEXEC (shebang-less file) fails the launch cleanly (documented `execvp`-fallback behavior change); **cancel_fd** — writing the shutdown byte during a hung handshake returns `failed_step = cancelled` with the child killed + reaped; **capture-binding** — a child that exits instantly (and whose pid the test races to reuse) never yields a record bound to the recycled occupant's identity (the parent captures pre-reap); **execute-only inputs** — an `--x`-only native binary and an `--x`-only script/interpreter still launch (EXEC_PATH plans need no readable fd; an EXEC_FD inspection failure degrades to EXEC_PATH-uncontained, not a launch failure); **probe seam** — forcing `pty_probe_execveat()→0` makes every launch EXEC_PATH-uncontained and runs the original path+argv unchanged (the `<3.19` fallback, no legacy kernel needed); **forced capture failure → daemon restart** — the marker scan reaps the prior-incarnation process and the `identity_unavailable` record self-deletes (Linux automated recovery, not manual); **leak loop** — a success/failure/cancel launch loop leaks no fds and no native heap (`pty_plan_free` on every path).
- **macOS:** Option A — the `mac:{bootsessionuuid}:{p_uniqueid}` token round-trips capture→record→compare exactly; **runtime capture smoke in the per-RID packaging checks** (capture own + child ids from the real kernel; non-zero, distinct — availability proven by execution, not mocks); the record pass reaps a survivor on an exact `mac:` match (the OS-aware tests' macOS branch flips from spare to reap); same-scheme **different** `p_uniqueid` (recycled pid) → conclusively gone, record deleted, occupant spared; **backward-compat deserialization** — the exact pre-M1-A JSON shape with `identity_kind` ABSENT (+ a nonempty legacy `start_identity`) deserializes as `present` (not quarantined), while a new inconsistent shape (`present`+empty / `identity_unavailable`+nonempty) → `.corrupt`; a **live legacy survivor behind a pre-M1-A `tk:` record** → spared every pass, classified `legacy_unresolvable`, logged loudly, record retained — then deleted normally once the pid is (manually) killed (the full upgrade contract, not a token-comparison unit); anomalous `proc_pidinfo` (EINVAL / short read / zero id) → an `identity_unavailable` record is persisted + reloaded (real round-trip, distinct from `.corrupt`), the launch proceeds, and the record classifies `identity_unresolvable` (manual-only on macOS; deletion after manual kill asserted); **crash-injection in the `forkpty`→record-rename window** → survivor is *not* recovered — the test documents the accepted recordless residual explicitly, so coverage claims can't drift.
