# AI-1313 ŌĆö Flow-reviewer capacity reaping: design spec (r14)

**Issue:** [AI-1313](https://linear.app/kurrent/issue/AI-1313) ŌĆö flow reviewers leak daemon agent-capacity: stale/superseded reviewers are never reaped ŌåÆ daemon hits 5/5 ŌåÆ all flows 400.
**Repos:** `kurrent-io/kcap-server` (reviewer lifecycle authority) + `kurrent-io/kcap-cli` (daemon self-defense). Two independently deployable PRs.
**Worktrees:** server `.claude/worktrees/ai-1313-reviewer-reaping` (off main @ `5bcf7787`); CLI `~/dev/kcap-cli-wt/ai-1313-reviewer-reaping` (off main @ `f082264`).

## 1. Problem

Observed 2026-07-10: after several `start_review_flow` attempts during an AI-1155 daemon crash-loop window, 5 reviewer processes ran 60ŌĆō91 minutes past any live round, filling the daemon's 5/5 agent capacity. Every subsequent flow failed `400 Daemon 'tony' is at max capacity (5/5 agents)` until the daemon was manually restarted.

Code verification (both repos, off main) found **five compounding defects**:

1. **The server's only kill path silently no-ops.** `AgentStoreDataService.RequestStopAgentAsync` (`AgentStoreDataService.cs:832-856`) looks up the in-memory `AgentInstanceRegistry`; if the entry is gone (server restart, reconcile-to-Failed, unregister) it returns without sending anything (`:835-837`). The daemon never hears `StopAgent`; the process keeps its slot.
2. **Terminal runs never durably stop participants.** `CloseAsync` / `FailRunAndStopAgentAsync` / the completion-runner terminal paths / idle-reap all route through `StopAllParticipantsAsync` (`FlowOrchestratorService.cs:451-463`), which only calls the best-effort in-memory stop ŌĆö it appends **no** `FlowParticipantStopped`. The projector's `HandleRunClosed`/`HandleRunFailed` (`FlowProjector.cs:494-512`) never set `flow_participants.stopped_at`. A leaked reviewer of a closed/failed run therefore remains an active-looking row (`stopped_at IS NULL`) ŌĆö invisible as a leak, but conveniently queryable.
3. **Both existing reapers gate on the wrong signal.** `OrphanedHostedAgentReaper` (`OrphanedHostedAgentReaper.cs:54-75`) requires the reviewer's **session** to be terminally ended ŌĆö a leaked idle reviewer's session never ends. `FlowReconcilerService.ReapIdleRunsAsync` (`FlowReconcilerService.cs:206-271`) only touches **open** runs (`status NOT IN ('closed','failed')`, `ReadModelQueries.cs:3250`) past a 24h idle TTL, and its stop is the same fragile registry-keyed path (defect 1).
4. **No dedup on `start_review_flow` + a daemon post-spawn leak.** `StartReviewFlowAsync` (`FlowOrchestratorService.cs:92-301`) has no dedup for catalog review kinds and the `MaxActiveRunsPerUser` cap applies only to dynamic flows (`:106-117`) ŌĆö a driver that lost its `flow_run_id` (context compaction) mints a brand-new reviewer per retry. Daemon-side, `HandleLaunchAgent`'s catch (`AgentOrchestrator.cs:563-605`) fires for exceptions after the PTY spawn (`:481`) + registry insert (`:515`) but before the read loop starts (`:552`) ŌĆö it cleans tokens/worktree but never terminates the process or removes it from `_agents`: an orphan that still counts toward `ActiveCount`, unreapable unless it stays silent 90s (`:1475-1478` covers `Starting` only).
5. **Daemon death makes orphans permanent.** Hosted agents are spawned via `forkpty` (`UnixPtyProcess.cs:54`) ŌĆö each becomes its own detached session leader. There is no `PR_SET_PDEATHSIG` (Linux/mac) and no Job Object (Windows, `ConPtyProcess.cs:204`), so **children survive a daemon SIGKILL** (static analysis; matches the live `ps` observation of 2026-07-10). The agent registry is memory-only (`AgentOrchestrator.cs:169`) and no PIDs are persisted, so a restarted daemon has no record of survivors and can never reap them.

Additional facts the design leans on:
- Capacity is enforced twice with the same drift-prone number: daemon-side `ActiveCount >= MaxConcurrentAgents` (`AgentOrchestrator.cs:283, 330-334`; default 5, `DaemonConfig.cs:7`), and server-side `daemon.ActiveAgents >= daemon.MaxAgents` in three launch guards (`CapacitorHub.cs:1003` interactive, `AgentStoreDataService.cs:676` shared internal launcher incl. review-flow, `:802` PR-review). The server counter (`DaemonRegistry.cs:293-306, 317`) increments on agent-registered and decrements **only** on daemon `AgentUnregistered` (`CapacitorHub.cs:1555-1556`).
- The durable identity needed for reaping already exists: `FlowRoleAgentAssigned` carries `flowRunId, role, agentId, sessionId, daemonName` (`FlowEvents.cs:32-53`), projected into `flow_participants` with a partial index on active rows (`CapacitorDb.cs:1716-1730, 1749-1750`).
- Reviewers are indistinguishable in the daemon registry after launch: `AgentInstance` retains no `Kind` and no flow identity (`AgentOrchestrator.cs:18-124`); `LaunchKind.ReviewFlow` (`Models.cs:1187-1191`) is used at launch time only. The child's environment does carry `KCAP_AGENT_ID` (and hosted-agent markers) set by `PtyHostedAgentRuntimeFactory` (`:77-96`).
- The AI-1185 anti-false-reap guards are deliberate and must be preserved: durable participant death is recorded **only** from confirmed-death seams, never from mere daemon absence (`HostedAgentDeathBridge.cs:5-22`). This design's new durable stops derive from **run-terminal facts** (closed/failed/superseded) ŌĆö unambiguous, so no new false-reap risk.
- **Single-process assumption.** kcap-server Phase 1 is single-process ŌĆö one owner of all writes (CLAUDE.md item 5). The per-owner start serialization (S4), the per-daemon dispatcher (S5), and the single reconciler instance (S3) rely on this; multi-replica coordination is explicitly deferred with the platform's Phase 2 (┬¦10).

## 2. Resolved unknown: reviewers survive daemon SIGKILL

The issue's open question is settled **statically**: all three OS mechanisms that would kill children on daemon death are absent (session-leader detachment via `forkpty`; no parent-death signal; no Job Object). Conclusion: reviewer processes survive. No live SIGKILL reproduction is required for this design (and none may be run against the real daemon ŌĆö other sessions actively use review flows); the orphan-kill machinery is verified in tests against isolated dummy processes (┬¦8). The design is correct under either outcome: startup orphan cleanup kills survivors if they exist and merely garbage-collects records if they don't.

## 3. Goals / non-goals

**Goals**
1. A reviewer whose run is closed/failed/superseded is stopped promptly and durably, releasing its capacity slot ŌĆö even when the in-memory registries were lost ŌĆö and the **physical kill is retried until the daemon confirms the process gone** (a durable "stopped" record never silences the reaper while the process lives, ┬¦5.3c).
2. `start_review_flow` retries stop piling up reviewers (supersede-same-target + per-owner cap, race-free under concurrent calls).
3. At-capacity `ReviewFlow` launches queue (FIFO per daemon, TTL, lifecycle-coupled) instead of failing.
4. The daemon self-defends: post-spawn launch failures clean up; reviewers have a lifetime/idle backstop; a restarted daemon kills orphans of previous incarnations. Containment: **absolute for the spawned agent process and its descendants on Windows (Job Object)**; **absolute for the spawned agent process on Linux (PDEATHSIG under the ┬¦6.4(1) thread-lifetime rules; records+scan stay active there as the descendant/backstop layer)**; on macOS the residual forkŌåÆrecord window is recovered by a marker scan retried every heartbeat ŌĆö unrecoverable **only** under a *permanent* same-user process-enumeration failure, which is logged loudly (┬¦6.4; the claim is deliberately honest, not absolute).
5. A 5/5 condition is diagnosable (what holds the slots, how old), with the **daemon-reported live list as the authority** when available.

**Non-goals**
- No separate reviewer capacity pool ŌĆö the shared `MaxConcurrentAgents` budget stays (decision).
- No queue persistence across server restarts (accepted degradation with a specified recovery contract, ┬¦5.5).
- No fix for the daemon crash-loop itself (AI-1155, separate) and no `FlowReviewerModelResolved` event (AI-1276 follow-up).
- No changes to interactive-agent launch UX (still fails fast at capacity).
- No multi-replica server coordination (Phase 2, ┬¦1 single-process assumption).

**Decisions (user-confirmed)**
- Scope: full fix, both repos.
- Capacity: shared budget kept; queue instead of reject for `ReviewFlow` launches.
- Pile-up: supersede same owner+kind+target, plus a per-owner active-run cap as backstop.
- Daemon backstop: reviewer max lifetime **6h** + idle timeout **2h**, both configurable.

## 4. Design overview

The **server** is the lifecycle authority: run-terminal facts on the `FlowRun-{id}` stream drive durable per-role `FlowParticipantStopped` events (the *logical* stop) while a separate liveness-reconciliation loop retries the *physical* kill until the daemon no longer reports the process (┬¦5.3). A sweep in the existing `FlowReconcilerService` catches anything missed ŌĆö including all historical leaks, retroactively, on first deploy. The **daemon** is self-defending: it contains children where the OS allows (Windows Job Object), persists child identity atomically at spawn with an environment-marker scan as the gap backstop (Unix), cleans up failed launches, bounds reviewer lifetime, and kills orphans of previous incarnations at startup ŌĆö so even a server that never says anything cannot leak slots forever.

**Logical vs physical stop (used throughout):** the durable `FlowParticipantStopped` event records the *decision* that a participant's service is over; it is terminal and append-once. Whether the *process* is actually dead is a separate, retryable concern driven by daemon-reported liveness ŌĆö the logical record never suppresses physical retry.

## 5. Server design (kcap-server)

### 5.1 S1 ŌĆö Durable stop on terminal paths (single seam)

`StopAllParticipantsAsync` (`FlowOrchestratorService.cs:451-463`) is the shared seam used by every terminal path (`CloseAsync:1067`, `FailRunAndStopAgentAsync:444`, `FlowRoundCompletionRunner.cs:132,211`, idle-reap `FlowReconcilerService.cs:254-266`). It gains a `reason` parameter and, **before** the best-effort gateway stop, appends `FlowParticipantStopped(flowRunId, role, agentId, reason)` for every assigned, non-stopped role. Reasons (closed set): `run_closed`, `run_failed`, `idle_reaped`, `superseded`, `stale_sweep`.

- **Ordering invariant (pinned):** in every S1 caller the run-terminal event (`FlowRunClosed`/`FlowRunFailed`) is appended **before** the `FlowParticipantStopped` events, so the durable stop is always justified by an already-durable run-terminal fact (preserving the AI-1185 rationale: stops derive from unambiguous lifecycle facts, never from daemon absence).
- **Append shape:** all of a run's `FlowParticipantStopped` events go in **one batch append** to `FlowRun-{id}` with a single CAS at the fold version (one stream ŌĆö no per-role version dance). On wrong-expected-version: **refold and re-decide** ŌĆö append only for roles still unstopped; if none remain, skip. On repeated conflict, log and continue to the gateway stop ŌĆö the sweep (S3a) re-drives the durable append later; the physical stop (S2) is **always** issued regardless of append outcome.
- **Queue coupling:** the same seam cancels any pending launch-queue tickets for the run (┬¦5.5) before stopping live agents, so a terminalized run can never launch a new reviewer afterwards.
- Fold semantics of `FlowParticipantStopped` are unchanged (sets `Stopped`, does not clear the in-flight round, does not advance `LastActivityAt` ŌĆö pinned in `FlowEvents.cs:126-127`). The run is terminal in every S1 caller, so marking roles stopped has no live-flow side effects.
- The existing projector handling of `FlowParticipantStopped` sets `flow_participants.stopped_at`/`stop_reason` ŌĆö the read model heals with no projector change and **no projection version bump**.

### 5.2 S2 ŌĆö Registry-independent stop

`RequestStopAgentAsync` (`AgentStoreDataService.cs:832-856`) gains an optional `daemonName` hint:

1. Registry entry present ŌåÆ current behavior (send `StopAgent` to `entry.ConnectionId` when reachable).
2. Entry missing and `daemonName` provided ŌåÆ resolve the daemon's **current** connection via `DaemonRegistry`; if connected, send `StopAgent(agentId)` anyway. The daemon handles an unknown agentId via its PID-record fallback (D4) ŌĆö or no-ops harmlessly on old daemons / already-dead processes.
3. Neither ŌåÆ log and return (unchanged).

**Routing rule (stale-entry safe):** the registry entry's `ConnectionId` is used only when it equals the daemon's **current** registered connection (per `DaemonRegistry`). If the entry is missing, its connection is stale (daemon reconnected since), the daemon is unreachable, **or the send fails**, the stop is routed via `daemonName` ŌåÆ `DaemonRegistry`'s current connection. A stale in-memory entry can therefore never pin retries to a dead connection (┬¦5.3(c) depends on this).

`FlowReviewerAgentGateway.StopAsync` gains the `daemonName` parameter; all flow callers have it durably (fold `FlowRoleAgentAssigned.DaemonName` / `flow_participants.daemon_name`). The send is best-effort and **repeatable**: delivery is not assumed ŌĆö ┬¦5.3(c) retries until the daemon's live report no longer contains the agent. The server does **not** decrement `ActiveAgents` on a fallback send ŌĆö the count corrects via the daemon's `AgentUnregistered` or the D2 status-report re-sync.

### 5.3 S3 ŌĆö Stale-reviewer reconciliation (FlowReconcilerService)

Three sub-passes in the existing 3-minute sweep (single reconciler instance ŌĆö ┬¦1 single-process assumption), after `ReapIdleRunsAsync`:

**(a) Logical repair ŌĆö participants of terminal runs.** Query (deterministic order):
`SELECT p.flow_run_id, p.role, p.agent_id, p.daemon_name FROM flow_participants p JOIN flow_runs r USING (flow_run_id) WHERE p.stopped_at IS NULL AND r.status IN ('closed','failed') ORDER BY p.assigned_at ASC LIMIT 50`.
Group rows by run; per run: fold `FlowRun-{id}` (authoritative). Roles already `Stopped` in the fold are **projector lag** ŌĆö do *not* re-append, but **do** issue the S2 stop for them (the physical kill is never skipped because the logical record exists). Roles unstopped in the fold get the batch `FlowParticipantStopped(reason: stale_sweep)` append (CAS; refold-and-re-decide on conflict, as S1) followed by the S2 stop. Oldest-first ordering + the fact that lagged rows resolve as the projector catches up bounds starvation; reaped/remaining counts are logged. **First deploy retroactively repairs every historical leak** (all leaked reviewers of closed/failed runs have `stopped_at IS NULL` today).

A "participant row present but run row missing" case is not handled specially: both rows are projected from the same `FlowRun-{id}` stream by the same projector, so the join covers all real states.

**(b) Untracked-reviewer cross-check.** The incident's worst case is a reviewer process with **no durable flow record** (crash between launch dispatch and the `FlowRoleAgentAssigned` append). Visible only via the daemon's live-agent report (D2, which carries the launch-time `FlowRunId`/`Role` ŌĆö ┬¦6.2): for each connected daemon reporting `LiveAgents`, an entry with `Kind == ReviewFlow` is reaped (S2 stop; no durable event ŌĆö there is no assigned run to record against) when **all** hold: (i) no active `flow_participants` row (`FindFlowByParticipantAgentAsync(agentId, includeTerminal: false)` is null); (ii) no live **launch reservation** for that agentId (┬¦5.5 ŌĆö covers the dispatchŌåÆregisterŌåÆassign window positively; mere presence in `AgentInstanceRegistry` is **not** an exemption, since the crash-between-launch-and-assign orphan sits in that registry indefinitely); (iii) daemon-reported `CreatedAt` older than a 15-minute grace ŌĆö **except** that the grace is bypassed and the reap is immediate when the entry's reported `(FlowRunId, Role)` is decisive: the fold shows that run terminal, or that role stopped or assigned to a *different* agentId. Skipped for old daemons that don't report `LiveAgents` (the D3 daemon TTL still bounds those).

**(c) Physical reconciliation ŌĆö retry-until-gone.** For each connected daemon reporting `LiveAgents`: any reported live agent whose flow participation is **logically over** ŌĆö its `flow_participants` row is stopped, or its run is terminal ŌĆö gets the S2 stop re-issued **every sweep until the daemon stops reporting it**. This is the leg that makes kills reliable under disconnects, dropped sends, and kill-resistant processes: the retry loop is keyed on daemon-reported liveness, never on `stopped_at IS NULL`, so appending the logical stop (which removes the row from pass (a)) cannot silence it. Escalation: after 3 consecutive sweeps still-live, log at warning with the slot-holder detail (S6).

### 5.4 S4 ŌĆö Supersede same-target + per-owner cap (StartReviewFlowAsync)

The supersede + cap + create decision runs **serialized per owner** (in-memory per-owner `SemaphoreSlim`, valid under the single-process assumption), inserted after auth/definition resolution, before the `FlowRunStarted` append:

1. **Supersede.** Resolve the owner's active runs via the event-store `FlowOwnerRuns-{ownerUserId}` index (AI-1277 2b; rebuild-immune). For each candidate (bounded by the cap), fold `FlowRun-{id}` and compare: same definition kind **and** same `(target_kind, trimmed target_ref)` exact match ŌćÆ supersede: CAS-append `FlowRunFailed(reason: "superseded")`, then `StopAllParticipantsAsync(reason: superseded)` (S1 seam ŌĆö durable stops + queue-ticket cancellation + S2 stops).
   - CAS conflict where the refold shows the run **already terminal** ŌåÆ benign, skip it and continue.
   - Any **other** supersede failure (transient store error, non-terminal conflict that persists after one refold+retry) ŌåÆ **abort the new start** with coded, retryable `409 flows_supersede_conflict` ŌĆö a failed supersede must not mint the very duplicate this feature prevents.
   - Old run terminalized but the new run's creation then fails ŌåÆ nothing minted; the supersede stands (the old run was a duplicate target by definition); the caller retries into a now-clean state.
   - Exact-match normalization is deliberate: the pile-up caller is the *same driver* retrying the same ref; PR-URLŌåönumber normalization is out of scope (┬¦10).
2. **Cap.** A new `Flows:MaxActiveRunsPerOwner` (default **10**) counts ALL of the owner's active runs (catalog + dynamic) via the same index, evaluated inside the same per-owner critical section (after the supersede pass) ŌåÆ at/over the cap: coded `400 flows_owner_cap_exceeded: close or supersede existing runs`. The existing stricter dynamic-only cap is unchanged. Because both the check and the `FlowRunStarted` append happen inside the lock, two concurrent same-owner starts cannot both pass a boundary snapshot (concurrent-start dedup and cap atomicity in one mechanism).
3. **Read-your-write across the lock (owner-index lag).** The `FlowOwnerRuns` index append is a separate, best-effort write after `FlowRunStarted`, so the index alone cannot guarantee the *next* serialized call sees the run just created. The decision therefore reads the index **merged with an in-memory per-owner recent-starts overlay**: on `FlowRunStarted` success (still inside the lock) the new run's `(runId, kind, target)` is added to the overlay; the dedup and cap checks consult index Ōł¬ overlay. An overlay entry is evicted once the owner index observably contains the run or the run's fold is terminal ŌĆö so a failed/lagging index append cannot let a duplicate slip through within an incarnation. **Across a restart** (overlay lost, index append never landed): S4 decisions are **gated on the reconciler's startup owner-index repair pass** ŌĆö the existing `FlowReconcilerService` startup reconcile, which rebuilds the owner index from the `FlowRun-*` scan, must complete before the first post-restart start is admitted (a readiness gate on the orchestrator; starts arriving earlier wait or return the retryable `server_catching_up` error the CLI already handles). A pre-restart `FlowRunStarted` whose index entry was lost is therefore repaired before any post-restart dedup/cap decision reads the index. The in-lock index append itself is synchronous with one retry (shrinking the window at the source).

### 5.5 S5 ŌĆö Launch queue + capacity reservations (ReviewFlow only)

Two cooperating in-memory singletons (single-process assumption):

**Launch reservations.** A per-daemon set of in-flight launches: reserved at dispatch (`agentId`, `flowRunId`, `role`, `dispatchedAt`), released on `AgentRegistered`, `LaunchFailed`, or a 90s dispatch timeout. **Effective capacity everywhere becomes `ActiveAgents + reservations >= MaxAgents`** ŌĆö all three existing guards (`CapacitorHub.cs:1003`, `AgentStoreDataService.cs:676`, `:802`) adopt it, closing the register-lag double-dispatch window. Reservations also feed S3(b)'s exemption (ii).

**DaemonLaunchQueue.** In `LaunchAgentInternalAsync`, a `Kind == ReviewFlow` launch at effective capacity **enqueues** instead of throwing; non-ReviewFlow launches keep the immediate (now reservation-aware) error.

- **Ticket state machine (complete):** `{key: (flowRunId, role), payload, enqueuedAt, state, agentId?}`. States and legal transitions:
  - `Pending ŌåÆ Dispatching` (dispatcher claim) ┬Ę `Pending ŌåÆ Expired` (TTL) ┬Ę `Pending ŌåÆ Cancelled`
  - `Dispatching ŌåÆ AwaitingRegistration` (launch sent; agentId minted at claim is the **correlation/generation token**) ┬Ę `Dispatching ŌåÆ Cancelled`
  - `AwaitingRegistration ŌåÆ Completed` (`AgentRegistered` for this agentId) ┬Ę `ŌåÆ Pending` (**daemon capacity rejection** ŌĆö requeued at the head, keeping the *original* `enqueuedAt`/TTL; not a failure) ┬Ę `ŌåÆ Failed` (`LaunchFailed` other than capacity, or the 90s registration timeout) ┬Ę `ŌåÆ Cancelled`
  All transitions are atomic claims (lock/`Interlocked`), so expiry, cancellation, dispatch, and daemon callbacks can never race into two outcomes. **TTL applies only in `Pending`** (once dispatched, the registration timeout governs).
- **Terminal generations leave tombstones ŌĆö a timeout or cancellation is uncertainty, not proof of death.** Every `Failed`/`Cancelled` transition of a dispatched generation records the agentId in a tombstone set and immediately issues an S2 `StopAgent(agentId)`. A **late `AgentRegistered` for a tombstoned agentId is not ignored**: the handler performs the normal atomic accounting (release the reservation if still held, increment `ActiveAgents` ŌĆö the process *is* live) and then immediately stops the agent; its eventual `AgentUnregistered` decrements normally, so counts always track reality. Other late messages for stale generations transition nothing. S3(c)'s liveness reconciliation remains the sweep-level backstop until the id is absent.
- **Generation settlement (the only tombstone-eviction proof ŌĆö no wall-clocks, no assumed durations).** Exactly **two** message types are sequenced: `LaunchAgentCommand` and `StopAgentV2`, carrying `{Epoch, Seq, CommandId}` (a per-command GUID). Everything else ŌĆö `RequestStatusReport`, `DaemonStatusReport`, `CommandRejected` ŌĆö is **out-of-band, unsequenced, read-only control traffic**: it consumes no `Seq`, mutates no agent state or watermark, is permitted while the sequenced lane is paused, and its snapshots identify `{Epoch, HighestAcceptedSeq, LastProcessedSeq}` ŌĆö so uncertainty recovery via the nudge can never itself be blocked by the uncertainty it resolves. Semantics, pinned:
  - **Contiguous-prefix watermark.** The daemon executes sequenced commands **strictly serially per epoch** (one ordered command lane), and `LastProcessedSeq` advances only when a command reaches a *terminal* outcome (executed, rejected-as-item, or errored) ŌĆö never on handler start. It is therefore by construction the highest **gap-free** processed prefix: `LastProcessedSeq Ōēź N` proves command N itself terminally processed.
  - **Acceptance and lane-item creation are one atomic operation.** Under a single lock, the daemon either advances `HighestAcceptedSeq` to N *and* commits a terminally-processable lane item for N, or does neither; if item creation fails after the counter was reserved (shutdown race, allocation failure), a **synthesized errored terminal item at N** is committed in its place so the contiguous watermark can always advance ŌĆö an advertised-accepted `Seq` with no processable item is impossible. Reconnect reads of the counters take the same lock. The synthesized item records `internal_error` in the identity cache **and emits `CommandRejected(reason: internal_error)`** so the server gets an immediate terminal outcome: a launch releases its reservation and requeues at admission head (transient local error ŌĆö not a round failure); a stop fails its confirmed-absence barrier with a distinguishable reason (execution is never claimed).
  - **Gap-free acceptance + serialized sending + duplicate identity.** The daemon accepts a sequenced command only when `Seq == HighestAcceptedSeq + 1` (exact-next). It keeps a **per-epoch identity cache** `(Seq ŌåÆ CommandId, state, outcome)` with a *defined retirement proof*: the server's single sender lane sends an out-of-band **`AckProcessedPrefix {Epoch, UpToSeq}`** once it has observed `LastProcessedSeq Ōēź UpToSeq`, and only then may the daemon retire entries `Ōēż UpToSeq`. **Receiver validation (same lock as the cache/watermarks):** the daemon honors an ack only when `Epoch` = current, `UpToSeq Ōēż LastProcessedSeq`, and `UpToSeq` > the last acknowledged prefix (monotonic); anything else ŌĆö over-ahead, regressing, stale-epoch, reconnect-delayed ŌĆö is rejected/logged without eviction. If the cache reaches its bound before an ack, the daemon **backpressures** (rejects further sequenced accepts with reason `backpressure`) ŌĆö it never evicts unacknowledged identity. **Backpressure recovery loop (no livelock):** on a `backpressure` rejection the sender completes the ack cycle ŌĆö obtain a report, send `AckProcessedPrefix`, and treat **acceptance of the next exact `Seq` as the confirmation** that capacity was released ŌĆö under bounded exponential backoff (1s ŌåÆ 15s) before the admission-head ticket re-dispatches; the waiting ticket holds no reservation (it reacquires at dispatch), so other work is not starved and a lost first ack simply retries the cycle. An exact duplicate (same `Epoch, Seq, CommandId`) is answered with an out-of-band **`CommandAck {Epoch, Seq, CommandId, State: Accepted|Processed, Outcome?}`** ŌĆö no re-execution. `Outcome` separates the **immutable execution fact** from **current liveness** (the fact never changes; liveness does ŌĆö conflating them would let a cached "live" resurrect a dead agent's accounting):
    - `LaunchExecuted {AgentId, SessionId?, ŌĆ”registration metadata}` + **`CurrentState: Live | Quarantined | Dead`** ŌĆö the execution fact is cached immutably; `CurrentState` is read **at ack time under the same per-agent lifecycle lock as the ┬¦6.1 move rule** (a single daemon-side lifecycle state machine ŌĆö never an unlocked probe of `_agents`/quarantine/records separately), with precedence `Live ŌåÆ Quarantined ŌåÆ Dead`, where `Dead` requires *confirmed* death ŌĆö a teardown racing the read can therefore never surface a transient false `Dead`. Server transitions: `Live` ŌåÆ apply the same atomic accounting as `AgentRegistered` (idempotent per agentId; ticket `Completes`); `Quarantined` ŌåÆ ┬¦6.1 unconfirmed-death rules (no live accounting, absence barriers unsatisfied); `Dead` ŌåÆ **historical acceptance only** ŌĆö no registry resurrection, no count increment, the generation settles. A retired-generation marker suppresses stale live-accounting from acks exactly as it does from delayed `AgentRegistered`.
    - `LaunchRejected {code}` ŌĆö per-reason semantics as `CommandRejected`.
    - `LaunchFailedCleaned` ŌĆö terminal, no process (confirmed dead at teardown): reservation released, ticket ŌåÆ `Failed` (existing `LaunchFailed` handling).
    - `StopExecuted` + `CurrentState: Dead | Quarantined | NotFound` ŌĆö `Dead` satisfies confirmed-absence barriers; `Quarantined` explicitly does **not** (┬¦6.1 governs); `NotFound` (no such agent in `_agents`/records/quarantine) combined with a settled watermark is absence evidence.
    - `InternalError` ŌĆö as the synthesized-item rule.
    `Accepted` (not yet processed) ŌåÆ the ticket stays `AwaitingRegistration`; nothing is inferred.
  - **One per-agent accounting state machine.** `AgentRegistered`, `AgentUnregistered`, and `CommandAck`-derived transitions all route through a single per-agent state machine under one accounting lock, with pinned precedence: **confirmed-dead/retired dominates** any later historical live signal (a late `LaunchExecuted`/`AgentRegistered` after observed death changes nothing); live-registration transitions are idempotent; unregister decrements exactly once and only if the live state was counted. Message arrival order (`AckŌåÆUnregistered`, `UnregisteredŌåÆAck`, `RegisteredŌåÆAckŌåÆUnregistered`, ŌĆ”) therefore cannot change the final state. A *different* `CommandId` at an already-accepted `Seq` is rejected as `duplicate_collision`. Server side, all sequenced sends go through **one serialized transport sequencer per daemon**, allocating `Seq` and transmitting in one critical section; after an *uncertain* send of N the sequencer pauses, nudges (out-of-band), and either observes N via `CommandAck`/watermark or retransmits the same `CommandId` at N ŌĆö the identity cache makes the retransmit-vs-delayed-original race harmless in both orders (including after N processed). `CommandId` is allocated once per ticket generation and **persists across transport retries for the ticket's lifetime**. The sequencer never allocates past an unresolved N.
  - **No blind replay ŌĆö semantic reconciliation before any fresh dispatch.** A `duplicate_collision` (or any state-loss ambiguity, e.g. after a server restart) is **never** resolved by automatically re-sending under a fresh `Seq`. The server first reconciles by *semantic identity*: check the agentId/flow generation against `LiveAgents Ōł¬ Quarantined` and the authoritative fold. **There is no adoption of unassigned survivors** (pinned, resolving any ambiguity with earlier wording): an unassigned live/quarantined survivor is always stopped via reap-before-respawn; "reuse" applies only to an agent **already durably assigned** in the fold (`FlowRoleAgentAssigned` exists for it) ŌĆö that is just the normal existing-participant path, no new event flow. Only when the prior process is provably absent does a launch re-enter admission. **At-most-once process creation per generation is the invariant.**
  - **Generation identity across outcomes (pinned).** Outcomes that *prove no process was created* ŌĆö `daemon_capacity`, `internal_error`, `wrong_next`, `duplicate_collision` resolved to provable absence, `backpressure` ŌĆö **preserve the semantic generation**: same agentId and `CommandId`, a new `Seq` on the next send (the daemon caches by `(Seq, CommandId)`, so the same `CommandId` at multiple `Seq`s is valid and is not replay). Recovery from *uncertain* outcomes (state loss, `stale_epoch` with a possibly-spawned process) mints **no new generation until absence is proven**, and then re-enters admission as a **fresh generation** (new agentId + `CommandId`).
  - **Rejected-but-viable launches re-enter through admission, never directly through the sequencer.** When a launch's rejection releases its reservation (`wrong_next`, `duplicate_collision`, `backpressure`, `internal_error`, `daemon_capacity`), the ticket returns to the **launch-capacity queue head** (original `enqueuedAt`/TTL/generation) and must re-validate the fold and **atomically reacquire capacity** before re-entering the transport sequencer ŌĆö a resync can therefore never send an unreserved launch past admission (no oversubscription, FIFO preserved). `StopAgentV2` needs no admission and resyncs directly.
  - **Transport sequencer ŌēĀ launch-capacity queue (two layers, no priority inversion).** The `DaemonLaunchQueue` (┬¦5.5 lifecycle/capacity admission) decides *when a launch becomes sendable*; only then does it enter the transport sequencer. Stops and control traffic **never wait behind an undispatchable launch**: a capacity-blocked ticket sits in the launch queue, not in the sequencer, so `StopAgentV2` (which is what frees capacity) and nudges flow immediately. Ordering is preserved among commands that have entered the sequencer.
  - **Allocation across server restarts.** The daemon reports `HighestAcceptedSeq` at `DaemonConnect` and in every report; a (re)connecting server resumes allocation at `HighestAcceptedSeq + 1`. `HighestAcceptedSeq` semantics, pinned: **epoch-wide** (one counter across all connections and sequenced command types), updated **atomically before the command is enqueued** into the serial execution lane (a reconnect can never observe a value lower than an already-accepted command), in-memory only (it legitimately dies with the epoch ŌĆö a new epoch is a new proof space), surviving SignalR reconnects within the same daemon process; `Int64`, non-wrapping (overflow is unreachable; treat wrap as protocol error).
  - **`CommandRejected {Epoch, Seq, CommandId, AgentId?, Reason}`** ŌĆö a new additive one-way daemonŌåÆserver message giving every rejection a server-visible terminal outcome, with **per-reason ticket semantics** (never a blanket failure):
    - `wrong_next` / `out_of_order` ŌĆö a transport-protocol condition from sender recovery: the sequencer pauses and resynchronizes (nudge ŌåÆ observe ŌåÆ retransmit); the launch ticket is *unaffected* (it never left its current state).
    - `duplicate_collision` ŌĆö a protocol invariant violation: alarm-level log + lane resync; the colliding command is re-sent under a fresh `Seq`; tickets unaffected.
    - `stale_epoch` ŌĆö **routes through the restart-heal barrier, never blind re-dispatch**: an uncertain launch may have been accepted and spawned by the dead epoch, so the ticket resolves into the heal path ŌĆö fold check + physical-liveness + boot-uncertainty (`StartupReapComplete`/positive-evidence) ŌĆö and is re-dispatched in the new epoch only when that barrier clears (a stale `StopAgentV2` similarly refreshes and retargets the new epoch's quarantine/records).
    - `daemon_capacity` ŌĆö requeue at head with original TTL (┬¦ ticket state machine; unchanged).
    - semantic launch rejection (invalid payload, unsupported vendor, ŌĆ”) ŌĆö terminal: the existing `LaunchFailed` path fails the round.
    Rejections never advance the old epoch's watermark; only the reservation tied to the attempted send is released where the reason implies no process can exist (`wrong_next`, `duplicate_collision`, `daemon_capacity`, semantic), while `stale_epoch` keeps uncertainty alive until the heal barrier resolves it.
  - **Settlement of generation `Seq = N`:** (a) its agentId appears (`AgentRegistered` ŌĆö the tombstone stop applies); or (b) a same-epoch report shows `LastProcessedSeq Ōēź N` with the agentId absent from the physical-liveness set (an instantaneous absence *before* the launch was processed can never evict ŌĆö closes the queued-command race); or (c) the daemon reconnects with a **new epoch** *and* the agentId is absent from the new incarnation's physical-liveness set ŌĆö which **includes boot-time uncertainty**: the new daemon reports all unresolved previous-incarnation candidates (leftover records + quarantine) in `Quarantined` from its very first `DaemonConnect`, plus a `StartupReapComplete` flag that turns true only once every record/marker candidate is confirmed dead and one full marker-scan pass has succeeded. While `StartupReapComplete` is false, discovery is incomplete, so **omission from a snapshot is not evidence**: an old-epoch generation settles only on **positive per-id absence evidence**, which has a concrete, durable shape:
    - `DaemonConnect`/`DaemonStatusReport` carry **`ConfirmedStartupCandidatesAbsent: [{Generation, AgentId, OldEpoch, FlowRunId?, FlowRole?}]`** ŌĆö every entry carries its own daemon-lifetime monotonic `Generation`, entries are reported in generation order, and the report includes `HighestResolutionGeneration` ŌĆö emitted for *every* confirmed-absence outcome: verified kill, already-exited (record identity matches no live process), or identity-mismatch (PID reused; the recorded agent is provably gone while the reused process is spared) ŌĆö and **never** for unreadable/ambiguous identity.
    - **Durable on the daemon side:** the resolved ledger is persisted in the daemon state dir (same atomic temp+rename discipline; the generation counter persists with it, spanning epochs and daemon restarts). Entries are re-reported at every connect/report until acknowledged.
    - **Sparse per-entry acknowledgement (no head-of-line retention):** `AckResolvedCandidates {Entries: [{Generation, AgentId, OldEpoch}]}` acknowledges **individual entries**, pruning each independently ŌĆö a long-unconsumable low-generation entry (open run awaiting its replacement) never blocks pruning of later consumed entries, so ledger and report stay bounded by the number of *genuinely open* proofs. An ack for an unknown/already-pruned key is idempotently ignored; a key never emitted is rejected/logged. Validation under the same lock as the ledger.
    - **Consumption is an outbox-ordered rule (atomic server side):** per entry ŌĆö (1) the durable flow decision commits first (`FlowRoleAgentAssigned` for the replacement, or the run-terminal event, observed via committed version/refold ŌĆö an in-flight append or in-memory fold is *not* consumption; entries with no flow identity are consumable immediately); (2) the server marks the entry consumed locally; (3) the ack is sent. A crash before (1) re-receives and re-evaluates; a crash after (1) but before (3) re-receives the entry, recognizes consumption **idempotently from the durable fold** (no repeated stop/relaunch side effects), and re-acks. A server crash before consumption therefore re-receives the evidence on reconnect; a daemon restart re-reports it from the persisted ledger (covers: kill in epoch E2, daemon restarts to E3 with discovery still incomplete ŌĆö the proof survives).
    A resolved entry settles exactly the matching old-epoch tombstones and unblocks exactly that role's heal barrier ŌĆö it authorizes nothing for unrelated ids. Mere absence from an incomplete set proves nothing (a recordless macOS gap-orphan can't be enumerated, so its absence proves nothing). Heal/relaunch for a role follows the same rule: positive per-id evidence or `StartupReapComplete`. Additionally, while `StartupReapComplete` is false, **unassigned-`ReviewFlow` healing is blocked for that daemon entirely when any unresolved candidate lacks flow identity** (a legacy or identity-less record can't be mapped to a role, so no role can be proven clear). On a macOS box with permanently failing enumeration this keeps failing retryably ŌĆö the honest ┬¦3 residual, never a blind relaunch.
  - **Retired-generation markers (event-order dedup).** After settlement via `AgentUnregistered`, a bounded retired-generation marker (agentId) is kept for a fixed 24h dedup window: a *delayed, older* `AgentRegistered` arriving after its own unregister is ignored for accounting (no increment) and answered with an idempotent stop. This is purely a count-hygiene guard (the process is dead); it prevents the permanent +1 drift that an old daemon (no re-sync reports) could otherwise accumulate.
  - A tombstone is removed **only on settlement**. There is deliberately **no time-based eviction** (`ReviewerMaxLifetime` is daemon-configurable and disable-able, so no server-side duration is a proof). Tombstones survive reconnects; if the set reaches its memory cap, **new dispatches backpressure** (the dispatcher pauses) rather than evicting an uncertain generation.
- **Reservation lifecycle (independent of ticket finality).** A reservation is created at send and released only by: (i) `AgentRegistered` ŌĆö atomic transfer to `ActiveAgents` (even for tombstoned generations, per the rule above); (ii) `LaunchFailed` for that agentId ŌĆö confirmed rejection; (iii) the bounded 90s unknown-outcome timeout ŌĆö capacity is released *and* the tombstone+stop machinery above owns the maybe-live process (a brief overshoot window is possible and self-corrects via the daemon-authoritative count re-sync, ┬¦6.2). Cancellation does **not** release a reservation early ŌĆö it rides (i)ŌĆō(iii).
- **Dedup:** enqueue is idempotent on the key ŌĆö a second enqueue while a ticket is in any non-final state returns the existing ticket.
- **Cancellation (linearized with dispatch):** S1's seam (`StopAllParticipantsAsync`) and S4's supersede cancel a run's tickets in **any non-final state**. For `Pending` that simply finalizes the ticket. For `Dispatching`/`AwaitingRegistration` ŌĆö where terminalization can race the send ŌĆö the ticket is marked `Cancelled` and the queue **kills the launch it can no longer prevent**: an S2 `StopAgent(agentId)` is issued immediately *and* re-issued from the `AgentRegistered` handler if the agent registers afterwards (the handler sees the final `Cancelled` state and stops the agent instead of completing). Belt-and-braces, the post-registration **assignment guard**: before appending `FlowRoleAgentAssigned`, the orchestrator re-folds the run ŌĆö if it went terminal, it does not assign and stops the agent. So a run terminalized at *any* point relative to dispatch ends with no unassigned survivor (and S3(b)'s decisive-identity reap is the sweep-level backstop).
- **Dispatch:** one serialized dispatcher per daemon (per-daemon lock) triggered by `AgentUnregistered` (slot freed), reservation release, and a 15s tick. It claims the head ticket (`Pending ŌåÆ Dispatching`), **re-validates against the authoritative fold** (run non-terminal, role unstopped ŌĆö else `Cancelled`), re-checks effective capacity, takes a reservation, sends the launch (`ŌåÆ AwaitingRegistration`).
- **Atomic reservationŌåÆcount transfer:** on `AgentRegistered`, the reservation removal and the `ActiveAgents` increment happen as one operation under the same per-daemon lock the admission checks take, **before** any drain trigger ŌĆö no window where a slot appears free to a concurrent drain.
- **TTL:** default 10 min (`Flows:LaunchQueueTtl`). Expiry (atomic `Pending ŌåÆ Expired`) synthesizes the existing daemon launch-failure path (`LaunchFailed`, coded `daemon_capacity_timeout`) so the round fast-fails exactly like a daemon-rejected launch ŌĆö no new failure plumbing.
- **Post-launch waits:** the reviewer session-id wait (`WaitForReviewerSessionIdAsync`, `FlowOrchestratorService.cs:1466`) starts its clock at **dispatch**; the overall bound is queue TTL + existing launch timeout.

**Restart recovery contract (accepted degradation, specified).** Queue and reservations are in-memory and vanish on server restart; recovery needs no new durable state:
- *Queued, never dispatched:* no process exists, no durable record exists. The in-flight round proceeds to its existing round timeout (definition `limits`) and fails, or the next `submit_review_round` relaunches the role (heal path, `FlowOrchestratorService.cs:725-764`) ŌĆö whichever comes first. Bound: the existing round timeout.
- *Dispatched just before the crash:* the daemon launches and registers the agent; no `FlowRoleAgentAssigned` was appended (server was down). The reviewer either completes and submits via the daemon-independent HTTP result path, or idles. Because `LiveAgents` carries the launch-time `(FlowRunId, Role)` (┬¦6.2), the restarted server is **not** blind to it: **reap-before-respawn, with confirmed absence** ŌĆö the heal path, before relaunching role R of run X, checks connected daemons' live reports for a `ReviewFlow` agent carrying `(X, R)` that the fold shows unassigned; any such survivor is S2-stopped and the heal path then **waits for confirmed absence before dispatching the replacement**. Confirmed absence is **sequence-barriered, never clock-compared**: the stop was sent with `Seq = M`; absence is confirmed by the daemon's `AgentUnregistered` for that agentId, or by a report for the same epoch with `LastProcessedSeq Ōēź M` omitting the id from the physical-liveness set (a pre-stop snapshot can therefore never be mistaken for confirmation, regardless of clock skew or reordering). The server sends a one-way `RequestStatusReport` nudge so confirmation is typically sub-second; the bounded wait is **90s** (one full periodic-report interval + jitter, so a lost `AgentUnregistered` alone cannot flake the wait). **Capability-gated:** this report-based mechanism exists only for D2-capable daemons (those that advertised `LiveAgents` at `DaemonConnect`). Against an **old daemon** ŌĆö which sends no reports at all ŌĆö the heal path never infers absence: it confirms only via `AgentUnregistered` within the wait, else returns the retryable error without relaunching. On timeout or an unreachable daemon it does **not** spawn concurrently ŌĆö it fails the submit with the existing retryable `participant_unreachable`-class error; S3(c) keeps retrying the kill and the next submit relaunches into a clean state. S3(b)'s decisive-identity reap (grace bypassed when `(FlowRunId, Role)` is contradicted by the fold) and D3's TTL are the sweep/daemon backstops. Net: a replacement is never launched while a same-role survivor is still confirmed-or-possibly live.

### 5.6 S6 ŌĆö Diagnosability (daemon-authoritative)

- **Authority:** when the daemon reports `LiveAgents` (D2), that list **is** the slot-holder truth. The server classifies each reported id by joining `flow_participants` (any status) + `AgentInstanceRegistry`: `reviewer (run X, role Y, age)`, `interactive (session Z, age)`, **`untracked (age)`** for ids matching nothing, plus the report's **`quarantined`** entries as their own class (┬¦6.4(2a)) ŌĆö the leak classes are first-class citizens, not omissions. A `server-count ŌēĀ daemon-count` disagreement is stated explicitly in the detail view. Fallback for old daemons: the server-side approximation (active participants + registry), labeled *approximate*.
- The three capacity error sites include an **aggregate** summary ŌĆö e.g. `at max capacity (5/5): 3 review-flow reviewers (oldest 47m), 1 interactive, 1 untracked` ŌĆö safe for any caller (no cross-user run ids). Full per-slot detail goes to server logs and the Agents tab daemon tooltip (run links only for runs the viewer can already see via existing visibility rules).

## 6. Daemon design (kcap-cli)

### 6.1 D1 ŌĆö Post-spawn launch-failure cleanup (single-flight)

`AgentInstance` gains an atomic `CleanupStarted` flag (`Interlocked.CompareExchange`); the launch-catch path and the read-loop `finally` both route through it, so exactly one teardown runs even when they race. Teardown in `HandleLaunchAgent`'s catch (when the `:515` insert happened): each step individually best-effort (own try/catch) ŌĆö `ReadCts.Cancel()`, `Runtime.TerminateAsync(5s)`, token revoke, worktree removal ŌĆö with `_agents.TryRemove` plus the `LaunchFailedAsync` + `AgentUnregisteredAsync` notifications in a `finally` so no step failure can suppress them or strand the registry entry. The PID record (D4) is deleted **only when termination is confirmed** (process observed exited); on unconfirmed termination the record is retained for the startup reaper. The pre-insert exception window (`:482-499`) is already safe.

**Unconfirmed-death teardown rule (applies to EVERY teardown path ŌĆö launch-catch, `HandleStopAgent`, read-loop cleanup, TTL reaps):** before an agent whose death is *not confirmed* is removed from `_agents`, its full launch metadata is **atomically moved into the kill-quarantine** (┬¦6.4(2a)). The agentId therefore never leaves the physical-liveness set (`LiveAgents Ōł¬ Quarantined`) while its OS process may still be alive: the stop command still *processes* (the watermark advances), but absence proofs are not satisfied ŌĆö settlement, heal confirmed-absence, and S3(c) all keep seeing the id until the heartbeat kill-retry confirms death. It also remains in `EffectiveCount`. Only a confirmed-dead agent exits both collections.

### 6.2 D2 ŌĆö Registry metadata + status report

- `AgentInstance` retains `Kind` (`LaunchKind`) plus the launch-time flow identity ŌĆö `LaunchAgentCommand` gains additive optional `FlowRunId`/`FlowRole` fields (set by the server for `ReviewFlow` launches) that the daemon stores on the instance. `CreatedAt`/`LastOutputAt` already exist.
- `DaemonConnect` gains an additive `LiveAgents: [{Id, Kind, CreatedAt, FlowRunId?, FlowRole?}]` alongside the existing `LiveAgentIds` (kept for back-compat). The flow identity is what lets a restarted server associate a surviving unassigned reviewer with its role (┬¦5.3(b) decisive reap, ┬¦5.5 reap-before-respawn) instead of waiting out a blind grace period.
- A new periodic (60s) daemonŌåÆserver `DaemonStatusReport { Epoch, LastProcessedSeq, ActiveCount, LiveAgents, Quarantined }` (┬¦6.4(2a) pins the `Quarantined` shape; `ActiveCount` = `_agents` Starting/Running only; `Epoch`/`LastProcessedSeq` implement ┬¦5.5 generation settlement ŌĆö serverŌåÆdaemon commands carry a per-epoch monotonic `Seq` the daemon acknowledges as fully-processed watermarks), sent via **one-way `SendAsync`** (never `InvokeAsync`) inside its own try/catch on the daemon ŌĆö with SignalR, an unknown hub method on an old server produces only a server-side log line and no client fault, and any send exception is logged without touching the agent loops. Server handler: accepts a report **only from the connection currently registered as that daemon** (stale-incarnation reports rejected by connection-id, the existing daemon-message pattern); within the current connection the daemon count is authoritative and overwrites `DaemonEntry.ActiveAgents`. A report racing a concurrent register/unregister on the same connection can be transiently off by one; the next report (Ōēż60s) corrects it, and admission uses reservations (┬¦5.5) on top ŌĆö bounded staleness, documented. Feeds S3(b)/(c) and S6.

### 6.3 D3 ŌĆö Reviewer lifetime/idle backstop

In the existing heartbeat loop (`RunHeartbeatLoopAsync`, `AgentOrchestrator.cs:1469`): for agents with `Kind == ReviewFlow && Status == "Running"`, reap via the normal `HandleStopAgent` path when `now ŌłÆ CreatedAt > ReviewerMaxLifetime` (default **6h**) or `now ŌłÆ LastOutputAt > ReviewerIdleTimeout` (default **2h**). End reasons `reviewer_ttl_expired` / `reviewer_idle_expired`. Config: `daemon.reviewer_max_lifetime` / `daemon.reviewer_idle_timeout` (+ env `KCAP_REVIEWER_MAX_LIFETIME` / `KCAP_REVIEWER_IDLE_TIMEOUT`); `0` disables. Interactive agents are untouched. Known caveat (documented): a TUI that repaints continuously keeps `LastOutputAt` fresh, so the idle clock may never fire ŌĆö the 6h lifetime is the hard bound. A false idle-kill self-heals: the next `submit_review_round` relaunches the role fresh (existing heal path).

### 6.4 D4 ŌĆö Child containment, durable PID records, and orphan reaping

Three layers, eliminating the unrecoverable window:

**(1) OS containment where available (Windows) ŌĆö atomic at creation.** `ConPtyProcess.Spawn` uses **creation-time job association**: a Job Object with `JOB_OBJECT_LIMIT_KILL_ON_JOB_CLOSE` is created first and passed to `CreateProcessW` via `STARTUPINFOEX` + `PROC_THREAD_ATTRIBUTE_JOB_LIST`, so the child is a job member **at the instant it exists** ŌĆö there is no post-create assignment window (no suspended-then-assign dance). Daemon death of any kind closes the job handle ŌåÆ the OS kills the child and its descendants. Every failure on the spawn path after `CreateProcessW` returns (attribute setup already done; record write, registration, anything) terminates the child via the job/`TerminateProcess` and **confirms death before returning**. On Windows the record layer below is thus only bookkeeping; there is no survivor class (job containment includes descendants). **Linux gets PDEATHSIG containment now** (promoted from follow-up), with its two sharp edges pinned:
- **Thread lifetime, enforced.** `PR_SET_PDEATHSIG` fires when the creating *thread* dies, not the process ŌĆö a thread-pool spawner would false-kill healthy reviewers on thread retirement, violating the anti-false-reap invariant. All Linux `forkpty` spawns therefore run on **one dedicated native spawner thread whose lifetime is exactly the daemon's lifetime** (created at daemon start, parked between spawns, never pooled). This invariant has a failure policy, not just an intention: **unexpected spawner-thread termination fail-fasts the whole daemon process** (children then die *with* the daemon ŌĆö exactly the semantic PDEATHSIG advertises ŌĆö instead of dying while a half-broken daemon keeps advertising itself); normal shutdown stops agents first, then retires the thread.
- **Death-before-prctl.** The child's post-fork sequence is `prctl(PR_SET_PDEATHSIG, SIGKILL)` ŌåÆ `if (getppid() != expectedParentPid) raise(SIGKILL)` ŌåÆ `exec`, where `expectedParentPid` is captured pre-fork ŌĆö a daemon that died in the forkŌåÆprctl window is detected by the reparent and the child self-kills. Because this is a post-fork path in a multithreaded process, the child executes **only an async-signal-safe native shim** between the fork return and `exec` ŌĆö prebuilt argv/envp, direct `prctl`/`getppid`/`raise`/`execve` ŌĆö never re-entering managed runtime code (which may hold locked runtime state at fork time).
- **Scope.** PDEATHSIG kills only the immediate child (the agent, a session leader); its own descendants normally exit on PTY/agent teardown, and the record+scan layers **stay active on Linux** as the descendant/edge-case backstop ŌĆö Linux containment is "absolute for the agent process", not "layers 2ŌĆō3 are bookkeeping".

**macOS has no OS primitive**, so it relies on layers 2ŌĆō3 with the honest residual stated in ┬¦3 goal 4: the forkŌåÆrecord-rename window is recovered by the marker scan, which runs at boot **and retries on every heartbeat tick** (not boot-only), so a *transient* enumeration failure merely delays recovery; only a *permanent* enumeration failure leaves the gap-orphan unrecovered, and that condition is logged loudly each cycle.

**(2) Atomic PID records with exact OS-native identity.** Immediately after `forkpty`/`CreateProcessW` returns ŌĆö before any await, registration, or other fallible step ŌĆö the daemon writes `<daemon-state-dir>/agents/{agentId}.json` via **temp-file + same-directory rename** (atomic; no fsync requirement ŌĆö a lost record falls to layer 3): `{agent_id, pid, start_identity, kind, vendor, flow_run_id?, flow_role?, daemon_id, daemon_epoch, spawned_at}`. **Process identity is the OS-native creation stamp, stored and compared exactly ŌĆö no tolerance:** Linux `/proc/{pid}/stat` `starttime` (clock ticks), macOS `proc_pidinfo`/`kinfo_proc` start `tv_sec/tv_usec`, Windows creation `FILETIME` ŌĆö captured in native units and never round-tripped through `DateTime` (this removes the PID-reuse false-kill a fuzzy timestamp comparison would reintroduce). On Unix, a record match must **additionally** show the expected `KCAP_AGENT_ID` in the process's environment before a kill; if identity or environment cannot be read, the process is **spared**, the record quarantined, and a warning logged ŌĆö ambiguity never kills. Deletion rules: a record is deleted **only after confirmed death** (exit observed, or the exact identity matches no live process) ŌĆö never after a merely *attempted* kill; a failed/timed-out kill retains the record for the next boot/sweep. Corrupt/unparseable records are quarantined (renamed `.corrupt`) and logged, never acted on.

**(2a) Record-write failure while the daemon lives.** The record write is itself the first fallible post-spawn step. If it fails (disk full, permissions, rename error), the launch **fails**: the daemon immediately performs emergency teardown using the runtime handle it still holds ŌĆö terminate ŌåÆ confirm death. If death is confirmed, it proceeds as an ordinary launch failure. If death **cannot** be confirmed, the full launch context `{agentId, pid, identity, kind, flowRunId?, flowRole?, createdAt}` enters an **in-memory kill-quarantine** the heartbeat loop retries (kill + confirm) until it succeeds. Integration is pinned: quarantined entries are reported in `DaemonStatusReport` as their own field (`Quarantined: [{Id, Kind, CreatedAt, FlowRunId?, FlowRole?}]`, same entry shape as `LiveAgents`), are **excluded from `ActiveCount`** (`ActiveCount` remains exactly the `_agents` Starting/Running count ŌĆö the wire field never changes meaning) **but included in the daemon's admission gate**: the launch guard becomes `EffectiveCount = ActiveCount + Quarantined.Count >= MaxConcurrentAgents` ŌĆö a persistent record-write/kill-failure mode therefore *fails closed* (admission shrinks by one per unconfirmed quarantined process; it can never mint unbounded processes beyond the configured budget). Both values are reported so the server can see *why* admission is blocked. Quarantined entries are classified as their own `quarantined` class in S6, and `HandleStopAgent(agentId)` consults the quarantine too ŌĆö so server-side S3(c) retries assist the heartbeat kill. Cleanup is never declared complete merely because the record could not be persisted.

**Physical-liveness set (used by every server-side liveness decision):** wherever this spec says a server pass consults the daemon's live report ŌĆö S3(b) untracked reap, S3(c) retry-until-gone, the restart reap-before-respawn and its confirmed-absence wait, and S6 authority ŌĆö the set is **`LiveAgents Ōł¬ Quarantined`**. Confirmed absence means the agentId is absent from **both** collections in a fresh snapshot. A quarantined reviewer carrying `(FlowRunId, Role)` is therefore visible to the heal path exactly like a live one, and no replacement launches while it exists.

**(2b) Record read-failure retry contract, and the two identity regimes.** Only *unparseable* records are renamed `.corrupt` (terminal, retained for the operator; documented as safe to delete once the marked process is confirmed gone). A record whose identity/environment verification fails **transiently** (proc/env read error) is left **in place** with a retry counter and re-verified on every subsequent heartbeat/startup pass ŌĆö it is never renamed for a transient failure, so the process it describes can neither become "recordless" to the marker scan nor be skipped forever. Kill authorization comes from exactly one of **two identity regimes**, pinned so implementers cannot diverge:
- **Record-based kill:** requires the exact native `(pid, start_identity)` match **plus** the Unix env `KCAP_AGENT_ID` check. Used by the startup record pass and the `StopAgent` fallback.
- **Marker-based kill:** requires reading the **full marker triple from the live process's own environment** ŌĆö `KCAP_AGENT_ID` present, `KCAP_DAEMON_ID` equal to this daemon's, `KCAP_DAEMON_EPOCH` from a *prior* incarnation. The triple was written by us at spawn and is read from the process itself, so it cannot be a PID-reuse artifact ŌĆö it **is** full identity for a recordless process; no native stamp is needed or possible. Used by the env-marker scan, including for a process whose record went `.corrupt` (a corrupt record does **not** suppress scan candidacy ŌĆö the marker regime simply takes over).
"A process is never killed solely because its record was quarantined" means: quarantining grants no kill authority by itself; one of the two regimes above must independently authorize at kill time.

**(3) Startup orphan reap + scoped env-marker backstop (Unix).** At daemon boot (next to the worktree `CleanupOrphanedAsync`, `DaemonRunner.cs:380`), under the daemon's existing exclusive state-dir lock (`DaemonLock`):
- For each leftover record **in this daemon's own state dir**: if the exact `(pid, start_identity)` matches a live process (and the Unix env check passes) ŌåÆ kill via **process group** (`killpg(pid, SIGTERM)` ŌĆö the `forkpty` child is a session leader so `pgid == pid`, taking its descendants too; fall back to `kill(pid)` if `killpg` errors) ŌåÆ 5s ŌåÆ `SIGKILL`; delete the record only on confirmed death. Identity-mismatched records are deleted (PID reuse / already dead). All kinds are reaped, not just reviewers ŌĆö an orphan of a dead daemon has lost its PTY master and is headless regardless.
- **Scoped env-marker scan (the record-gap backstop).** The child environment gains two markers set at spawn: `KCAP_DAEMON_ID` (a stable identity of this logical daemon = hash of its state-dir path) and `KCAP_DAEMON_EPOCH` (a fresh GUID per daemon boot, also persisted in the state dir). The scan enumerates same-user processes and kills (same procedure) only those whose environment carries `KCAP_AGENT_ID` **and** `KCAP_DAEMON_ID` equal to *this* daemon's id **and** `KCAP_DAEMON_EPOCH` different from the *current* incarnation's epoch, and which match no just-processed record. Agents of a **different** logical daemon under the same user are therefore never touched (their `KCAP_DAEMON_ID` differs), and the current incarnation's own children are protected by the epoch. Enumeration mechanism, pinned: Linux reads `/proc/{pid}/environ` (same-uid readable); macOS uses **one** mechanism ŌĆö `ps -E -ww -o pid=,command=` over the user's processes, parsing appended env ŌĆö and if enumeration fails or env is unreadable the scan **degrades to a logged no-op for that pass and retries on the next heartbeat tick** ŌĆö the scan is not boot-only, so a transient enumeration failure only delays gap recovery (per ┬¦3 goal 4, only a *permanent* enumeration failure leaves a macOS gap-orphan unrecovered, logged loudly each cycle). Windows and Linux need no scan ŌĆö layer 1 guarantees no survivors there.
- **StopAgent fallback:** `HandleStopAgent` for an agentId not in `_agents` consults the PID record and kills by exact `(pid, start_identity)` (+ Unix env check) ŌĆö this makes the server's S2 fallback effective against a *new* daemon incarnation that never knew the agent. Record deletion follows the confirmed-death rule.

## 7. Protocol & compatibility

All protocol changes are additive, with each affected message's wire shape pinned:
- `DaemonConnect` (existing DTO): gains optional `LiveAgents`, `Quarantined`, `HighestAcceptedSeq`, `LastProcessedSeq`, `Epoch`, `StartupReapComplete`, and a `SupportsSequencedCommands` capability flag. Old servers ignore unknown JSON fields; old daemons never set them.
- `LaunchAgentCommand` (existing DTO): gains optional `FlowRunId`, `FlowRole`, `Epoch`, `Seq`. Old daemons ignore unknown fields **and process the command on the legacy unsequenced path** (they don't advertise the capability, so the server never relies on sequenced semantics for them). A new daemon receiving a command **without** `Seq` (old server) processes it on the unsequenced lane and never advances the sequenced watermark.
- `StopAgent`: the legacy primitive `StopAgent(agentId)` is **unchanged** for old daemons. Capability-advertising daemons receive a new message `StopAgentV2 { AgentId, Epoch, Seq }` instead; the old primitive remains supported on new daemons (unsequenced lane) for old servers.
- `DaemonStatusReport` / `RequestStatusReport` / `CommandRejected` / `CommandAck` / `AckProcessedPrefix` / `AckResolvedCandidates`: new one-way, out-of-band messages; an unknown method on the receiving side is a server/daemon-side log line only (┬¦6.2). `DaemonConnect`/`DaemonStatusReport` additionally carry `ResolvedStartupCandidates` (┬¦5.5 positive per-id death evidence).
- **Settlement is capability-gated on `SupportsSequencedCommands`** ŌĆö not on `LiveAgents`/D2 alone. Against a non-capable daemon there are no watermarks; tombstones for such daemons settle only via `AgentRegistered`/`AgentUnregistered` or epoch change with specific-id absence, and heal confirmed-absence degrades to the old-daemon rule (┬¦5.5). Sequenced commands carry `Epoch` so a stale-incarnation command is rejected by the daemon rather than processed under a new epoch's proof space.
Compatibility matrix: old daemon + new server ŌĆö S3(b)/(c), re-sync, settlement, and the nudge are all skipped/degraded as specified; S1/S3(a)/S4/S5/S6-fallback work. New daemon + old server ŌĆö reports/nudges land on no handler (log only); unsequenced commands run on the legacy lane. No read-model projection version bump anywhere (S1 uses existing events; S6 adds queries only). No new tables; queue, reservations, tombstones, and slot-holder aggregation are in-memory/query-time. Old daemon + new server: S3(b)/(c) and count re-sync are skipped (no `LiveAgents`); S1/S3(a)/S4/S5/S6-fallback all work. New daemon + old server: the status report lands on no handler (server log only). No read-model projection version bump anywhere (S1 uses existing events; S6 adds queries only). No new tables; queue, reservations, and slot-holder aggregation are in-memory/query-time.
